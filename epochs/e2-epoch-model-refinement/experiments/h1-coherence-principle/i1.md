# H1 / I1

**Question:** What makes a set of experiments coherent enough to share an epoch?

**Analysis:**

The conjunctive model (A AND B AND C must all be true) is a special case, not the
general case. It works when the epoch-level decision is binary and depends on all
experiments passing. But most product development doesn't work that way — you batch
work because it's timely, related, or shares context, not because the outcomes are
logically entangled.

Three valid coherence principles, in decreasing strictness:

1. **Conjunctive proposition** — epoch closes only if all experiments confirm.
   Use when: a single architectural decision depends on multiple unknowns simultaneously.
   Example: "is semantic validation worth building?" requires judge accuracy AND
   acceptable latency AND workable authoring model.

2. **Thematic area** — experiments share a concern, resolve independently.
   Use when: a domain needs exploration across multiple independent questions.
   Example: "refine the epoch model" — H1, H2, H3, H4 are all about epochs but
   don't depend on each other's outcomes.

3. **Work batch** — experiments are grouped by timing or convenience.
   Use when: small independent improvements that don't merit their own epochs.
   Example: "fix soft warning copy + fix topic/goal naming + update README TCs"

In practice, distinguishing 2 and 3 isn't important — both are thematic batches.
What matters is that the epoch has a stated theme so `decision.md` can be written
coherently. The grouping principle doesn't need to be declared upfront.

**Verdict:** CONFIRMED. Thematic area is the right default. Conjunctive dependency
is opt-in — experiments can note it in their spec ("depends on H1 closing first")
if needed, but the default is independent.
