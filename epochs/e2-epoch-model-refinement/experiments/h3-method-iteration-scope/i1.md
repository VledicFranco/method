# H3 / I1

**Assessment:** Does method-iteration.yaml's 6-phase structure map to experiment-level work?

Phase 0 — Context Loading: inventory the product state and open epoch criteria.
  → Maps cleanly to "what's the state of this experiment's concern area?"

Phase 1 — Objective Setting: Be/Do/Motor + change_type + acceptance criteria.
  → Maps cleanly to experiment-level objective definition.

Phase 2 — Design: spec or technical design before touching files.
  → Maps cleanly to experiment design (methodology spec, technical design).

Phase 3 — Implementation: write the YAML/code.
  → Maps cleanly to experiment execution.

Phase 4 — Validation: run real sessions, log outcomes.
  → Maps cleanly to experiment validation.

Phase 5 — Decide: stop/continue with rationale.
  → Maps to experiment decision. "Stop" = write decision.md and close experiment.
     "Continue" = open i2.md and run another session.

All 6 phases map to experiment-level work without awkwardness. There is no phase
that naturally belongs at the epoch level — epoch opening/closing is not structured
work, it's document authoring (hypothesis.md, decision.md).

**One note:** Phase 0 asks for `epoch_criteria_open`. In the new model, this is
the experiment's own acceptance criteria, not the epoch's. The field name is slightly
misleading but not wrong — the experiment's criteria are derived from the epoch's theme.
Worth a wording fix in the guidance text, but not a structural change.

**Verdict:** CONFIRMED. method-iteration.yaml needs no structural changes.
Minor wording fix: Phase 0 guidance should say "experiment's acceptance criteria"
not "epoch's acceptance criteria" — but this is guidance text polish, not a
methodology change.
