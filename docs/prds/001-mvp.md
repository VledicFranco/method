**Status:** Refined
**Date:** 2026-03-14
**Previous:** Draft (2026-03-13)
**Scope:** MCP server that loads, serves, and guides agents through compiled methodologies

---

## Purpose

`pv-method` is the automation layer for the Voss‚ÄìSolariƒá methodology system. It:

1. **Serves** ‚Äî provides compiled methodology content (YAMLs, theory) to LLM agents via MCP
2. **Guides** ‚Äî gives agents step-by-step procedural guidance as they execute a method
3. **Installs** *(deferred)* ‚Äî defines how methodologies embed into a consuming project via `.methodology/`

The MVP proves concepts 1 and 2 end-to-end with any compiled method in the registry and in-memory state. Success criteria use P0-META/M1-MDES as the reference case.

---

## Problem

LLM agents executing formal methods (M1-IMPL, M1-MDES, etc.) currently:
- Load method documents manually each session from `ov-oss/methods/`
- Have no structured way to query "what step am I on?" or "what does this term mean?"
- Cannot persist execution state across conversations
- Have no canonical way to know which methodology applies to their project

The MCP server solves the first two for MVP. Persistence and project-scoping are post-MVP.

## MVP Scope

### In

| # | Capability | Details |
|---|-----------|---------|
| 1 | List methodologies | Scan `registry/`, return all available methodologies and their methods with descriptions |
| 2 | Load methodology | Given a methodology ID + method ID, parse the compiled YAML from `registry/` and initialize in-memory state |
| 3 | Query current step | Return the full step record: name, pre-conditions, guidance, output schema |
| 4 | Advance step | Mark current step complete and move to the next in linear order |
| 5 | Methodology status | Return what is loaded, which step is current, overall progress (n of total steps) |
| 6 | Theory lookup | Given a term or concept name, search `theory/F1-FTH.md` and `theory/F4-PHI.md` and return the relevant definition |

### Out of MVP

- `.methodology/` directory ‚Äî install mechanism deferred
- CLI (`method install`, `method status`) ‚Äî deferred
- Persistence across sessions ‚Äî in-memory only; agent re-loads each session
- Multiple concurrent methodology contexts
- DAG branching and loop edges (e.g., M1-IMPL's œÉ_A4 ‚Üí œÉ_A1 re-entry loop) ‚Äî MVP treats step lists as linear sequences; DAG-aware traversal is post-MVP
- Step validation against pre/post conditions ‚Äî guidance only, no enforcement
- Methodology-level transition function evaluation (Œ¥_Œ¶) ‚Äî MVP serves methods, not methodology-level routing

### Package responsibilities

**`@method/core`** ‚Äî owns all domain logic, no MCP dependency:
- `listMethodologies()` ‚Äî scan `registry/`, return available methodologies and methods with descriptions
- `loadMethodology(methodologyId, methodId)` ‚Äî parse YAML from `registry/`, extract step list
- `getCurrentStep()` ‚Äî return current step record
- `advanceStep()` ‚Äî move to next step in linear order
- `getStatus()` ‚Äî return loaded context + progress
- `lookupTheory(term)` ‚Äî search theory files by section heading and keyword, return relevant excerpt

**`@method/mcp`** ‚Äî thin MCP adapter, owns no domain logic:
- One file: `index.ts`
- Imports from `@method/core`
- Maps six core operations to six MCP tools
- Handles MCP protocol, input validation (Zod), error formatting

## MCP Tool Catalog

All tools return errors as `{ error: string }` when preconditions are not met.

### `methodology_list`
Discover all available methodologies and their methods. Scans the `registry/` directory, reads the top-level keys from each YAML (id, name, description), and returns a tree of what's loadable. This is the agent's entry point ‚Äî it calls `methodology_list` first to see what's available, then picks a method based on the descriptions.

```
Input:  {}
Output: [
  {
    methodology_id, name, description,
    methods: [{ method_id, name, description, step_count }]
  }
]
```

**Discovery flow:** The agent reads the list, matches its task to a method description, and calls `methodology_load`. The descriptions come from each YAML's `navigation.what` or top-level `description` field ‚Äî enough for an LLM to route correctly. Post-MVP, the server could expose methodology-level transition functions (Œ¥_Œ¶) as evaluable routing logic, but for the MVP, agent judgment on descriptions is sufficient.

### `methodology_load`
Load a method from a methodology into the active session context. Resets any existing session.

The `methodology_id` identifies the parent methodology directory in `registry/` (e.g., `P0-META`). The `method_id` identifies the specific method whose step DAG will be traversed (e.g., `M1-MDES`). The server reads the method's compiled YAML, extracts its `phases:` array as the step list, and initializes the session at step 0.

```
Input:  { methodology_id: string, method_id: string }
Output: { methodology_id, method_id, method_name, step_count, objective }
Error:  methodology_id or method_id not found in registry; YAML parse failure
```

### `methodology_status`
Return the current loaded context and execution progress.

```
Input:  {}
Output: { methodology_id, method_id, current_step_id, current_step_name, step_index, total_steps }
Error:  No methodology loaded
```

### `step_current`
Return the full record for the current step.

```
Input:  {}
Output: { id, name, precondition, postcondition, guidance, output_schema }
Error:  No methodology loaded
```

### `step_advance`
Mark the current step complete and advance to the next.

```
Input:  {}
Output: { previous_step, next_step } | { message: "Method complete" }
Error:  No methodology loaded; already at terminal step
```

### `theory_lookup`
Search theory files for a term or concept and return the relevant definition. Searches section headings and definition labels in `theory/F1-FTH.md` and `theory/F4-PHI.md`. Returns the matching section content. If no exact heading match, falls back to keyword search and returns the most relevant excerpt.

```
Input:  { term: string }
Output: { source, section, content } ‚Äî one or more matching excerpts
Error:  No matches found
```

## State Model (In-Memory)

```typescript
type Step = {
  id: string;                // e.g., "sigma_0"
  name: string;              // e.g., "Setup"
  role: string;              // e.g., "rho_leader"
  precondition: string;
  postcondition: string;
  guidance: string;          // full guidance text from YAML
  output_schema: object | null;
};

type SessionState = {
  loaded: boolean;
  methodologyId: string | null;
  methodId: string | null;
  methodName: string | null;
  steps: Step[];             // ordered step array extracted from YAML phases
  currentIndex: number;      // index into steps[]
};
```

State is initialized at server start (empty). Reset when `methodology_load` is called. Not persisted between server restarts.

**YAML mapping:** The loader reads the method YAML's `phases:` array. Each phase entry maps to a `Step`: `id` ‚Üê phase `id`, `name` ‚Üê phase `name`, `role` ‚Üê phase `role`, `precondition` ‚Üê phase `precondition`, `postcondition` ‚Üê phase `postcondition`, `guidance` ‚Üê phase `guidance`, `output_schema` ‚Üê phase `output_schema`. The loader must handle missing optional fields defensively (some phases omit `output_schema` or `role`).

**Linear traversal note:** The `currentIndex` pointer advances linearly through the `steps[]` array. This correctly handles path-graph DAGs (which all registry methods use at MVP scope). Methods with loop edges or branching (e.g., M1-IMPL's œÉ_A4 ‚Üí œÉ_A1) will need a DAG-aware traversal engine post-MVP.

Post-MVP: state will be read from / written to `.methodology/state` in the consuming project's CWD.

## YAML Loading

The loader resolves registry paths as: `registry/{methodology_id}/{method_id}/{method_id}.yaml`. For example, `methodology_load("P0-META", "M1-MDES")` reads `registry/P0-META/M1-MDES/M1-MDES.yaml`.

**Methods without sub-directories** (e.g., `P0-META` itself is a methodology-level YAML at `registry/P0-META/P0-META.yaml`) are also loadable: when `methodology_id === method_id`, or when the method_id path doesn't exist, fall back to `registry/{methodology_id}/{methodology_id}.yaml`.

The loader must:
- Parse the YAML and extract the `phases:` array
- Validate that at least one phase exists
- Extract each phase's required fields (`id`, `name`) and optional fields (`role`, `precondition`, `postcondition`, `guidance`, `output_schema`)
- Return a clear error if the file doesn't exist or lacks a `phases:` key

**Dependency:** Add `yaml` (or `js-yaml`) to `@method/core` for YAML parsing.

## Success Criteria

An agent connected to the MCP server can:

1. Call `methodology_load({ methodology_id: "P0-META", method_id: "M1-MDES" })` and get a confirmation with the method name and step count (7 steps for M1-MDES)
2. Call `step_current` and receive the full step record for œÉ_0 including its guidance text
3. Call `step_advance` three times and land on œÉ_3; each advance returns the previous and next step IDs
4. Call `methodology_status` and see `{ step_index: 3, total_steps: 7 }`
5. Call `theory_lookup({ term: "domain retraction" })` and receive the Definition 6.3 content from F1-FTH.md
6. Call `step_advance` at the terminal step and get an error
7. Call `methodology_status` with nothing loaded and get an error
8. Call `methodology_load({ methodology_id: "P1-EXEC", method_id: "M1-COUNCIL" })` and get 4 steps (œÉ_0 through œÉ_3) ‚Äî confirming the server works with any compiled method, not just P0-META
