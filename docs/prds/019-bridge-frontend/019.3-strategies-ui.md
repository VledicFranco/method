## 1. Purpose and Problem Statement

### Strategy information is fragmented across four surfaces

Today, strategy-related information lives in four disconnected places:

1. **Strategy YAML files** in `.method/strategies/` — the human reads raw YAML to understand what a strategy does, what nodes it contains, what triggers it declares, and what oversight rules apply. No rendered view exists.
2. **Strategy visualizer** at `GET /viz/*` — a standalone React SPA that shows DAG execution state with xyflow. It requires knowing an execution ID, has no discovery or browsing capability, and is not connected to the dashboard.
3. **Dashboard trigger panel** — shows trigger fire history in a table, but doesn't link to the strategy definition or execution visualization.
4. **Strategy retrospectives** in `.method/retros/` — YAML files with timing, cost, and gate results. No way to browse or compare them from the UI.

A human who wants to answer "what strategies exist, how are they configured, how did they perform, and how does this execution compare to the last one?" must visit four different surfaces, manually correlate IDs, and read raw YAML. This defeats the Narrative Flow principle of progressive disclosure.

### What this PRD delivers

A unified **Strategies** page within the Narrative Flow frontend that combines:

- **Strategy definition browser** — cards for each strategy in `.method/strategies/` with structured rendering (not raw YAML), mini-DAG thumbnails, trigger badges, and last execution summary.
- **Strategy detail view** — slide-over panel showing full strategy configuration, node list, trigger config, oversight rules, and gates — all rendered structurally from the parsed YAML.
- **DAG visualizer** — the existing xyflow canvas (`packages/bridge/viz/`) absorbed into the new frontend as a reusable page component, replacing the standalone SPA.
- **Live execution view** — real-time node status, cost tracking, and gate results displayed on the DAG canvas via SSE or polling.
- **Execution history** — chronological timeline of past executions with timing, cost, gate pass/fail, and retro links.
- **Execution comparison** — side-by-side view of two executions showing node-by-node timing, cost, and gate differences.
- **Gantt timeline** — parallel execution timing visualization showing which nodes ran concurrently and the critical path.
- **Execute button** — fire a strategy manually from the UI.

After this PRD, `packages/bridge/viz/` is deprecated and its `GET /viz/*` routes are removed. All strategy visualization happens within the Narrative Flow frontend.

## 2. Components

### Component 1: Strategy Definitions Endpoint

**New endpoint:** `GET /api/strategies/definitions`

Returns parsed strategy definitions from `.method/strategies/`, providing the data the frontend needs to render definition cards without requiring the client to parse YAML.

```typescript
interface StrategyDefinition {
  id: string;
  name: string;
  version: string;
  file_path: string;                     // Relative path within .method/strategies/
  triggers: Array<{
    type: string;                        // manual | git_commit | file_watch | schedule | webhook | ...
    config: Record<string, unknown>;     // Trigger-type-specific fields
  }>;
  nodes: Array<{
    id: string;
    type: 'methodology' | 'script';
    methodology?: string;                // For methodology nodes
    method_hint?: string;
    depends_on: string[];
    inputs: string[];
    outputs: string[];
    gates: Array<{
      type: string;
      check: string;
      max_retries: number;
    }>;
  }>;
  strategy_gates: Array<{
    id: string;
    depends_on: string[];
    type: string;
    check: string;
  }>;
  oversight_rules: Array<{
    condition: string;
    action: string;
  }>;
  context_inputs: Array<{
    name: string;
    type: string;
    default?: unknown;
  }>;
  outputs: Array<{
    type: string;
    target: string;
  }>;
  last_execution?: {                     // Most recent execution summary (if any)
    execution_id: string;
    status: string;
    cost_usd: number;
    duration_ms: number;
    completed_at: string;
    gates_passed: number;
    gates_failed: number;
  };
}
```

**Response:** `{ definitions: StrategyDefinition[] }`

**Implementation:** The endpoint reads and parses all `.yaml` files in the configured strategies directory (`TRIGGERS_STRATEGY_DIR`, default `.method/strategies`). Each file is parsed with `js-yaml` (per DR-05), validated for basic structural integrity, and cross-referenced against the in-memory execution store for last execution data. Files that fail to parse are included with an `error` field instead of the full definition.

**Caching:** Definitions are re-parsed on each request. The strategies directory is small (expected < 20 files), and parsing is fast (< 10ms total). If the directory grows, add a file-mtime-based cache.

### Component 2: Strategy Definition List Page

The primary Strategies page, accessible via the `Strategies` nav item.

**Layout:** Two sections within the 820px content area.

**Section 1 — Strategy Definitions (top)**

A 2-column grid of strategy cards (matching the [visual mock](ui-mock-strategies.html)). Each card shows:

- **Strategy ID** — mono, `--bio` color (e.g., `S-SMOKE-001`)
- **Strategy name** — Space Grotesk, bold (e.g., "AG-045 Smoke Test")
- **Mini-DAG thumbnail** — inline node/edge/gate visualization using colored dots and lines. Nodes are 10px circles (methodology = `--nebular`, script = `--bio`), edges are 20px lines, gates are 8px rotated squares. This is a CSS-only rendering — no canvas, no xyflow dependency for thumbnails.
- **Node type badges** — pill badges showing node types present (methodology, script)
- **Gate badges** — pill badges showing strategy gate IDs
- **Trigger badges** — pill badges showing trigger types (manual, git_commit, file_watch, etc.) with type-specific coloring per the mock
- **Last execution summary** — time since last run, cost, status badge (completed/failed)

**Card states:**
- Default: `--abyss` background, `--border` border
- Hover: lift 2px, border brightens to `--border-hover`, subtle shadow
- Selected: `--bio` border with 1px outer glow, opens slide-over detail panel
- Running: pulse-glow animation (2.5s ease-in-out) on border

**Card ordering:**
1. Currently running strategies (top, with glow)
2. Recently completed (sorted by last execution time, descending)
3. Never executed / definition-only (bottom)

**Section 2 — Execution History (below cards)**

A vertical timeline showing past executions across all strategies. Each timeline entry shows:
- Colored dot: `--cyan` for completed, `--error` for failed, `--bio` for running
- Timestamp (mono, dim)
- Strategy ID (mono, `--bio`)
- Status pill (completed/failed/running)
- Cost (mono)
- Duration (mono)
- Gate summary (e.g., "all gates passed", "1 gate retry", "gate exhausted retries")
- "Open in Visualizer" link (navigates to the execution's DAG view)

Timeline follows the Narrative Flow design language: vertical line on left (2px, `--border`), staggered entrance animation (100ms per item), grouped by time period.

**Data source:** Combines `GET /strategies` (execution list) with `GET /api/strategies/definitions` (definition metadata).

**Polling:** 5-second interval for execution list updates. Strategy definitions are fetched once on page load and on manual refresh.

### Component 3: Strategy Detail Panel (Slide-Over)

Clicking a strategy card opens a 420px slide-over panel from the right (per Narrative Flow layout spec). The panel renders the strategy definition structurally — not raw YAML.

**Header:**
- Type label: `STRATEGY DEFINITION` (mono, muted, uppercase)
- Strategy name (Space Grotesk, `--text-lg`)
- Close button (X)

**Tabs:**
- **Overview** — structured field rendering (as shown in mock)
- **YAML** — raw YAML with syntax highlighting (for power users who want the source)
- **History** — execution history filtered to this strategy

**Overview tab sections:**

1. **Identity** — ID, name, version as field-value rows
2. **DAG Nodes** — list of node cards, each showing:
   - Node type dot (color-coded) + node ID (mono, bold) + type label badge
   - Indented metadata: methodology/method_hint (for methodology nodes), depends_on, inputs, outputs, gate config
3. **Strategy Gates** — list of gate cards with diamond icon, gate ID, check expression, and depends_on
4. **Triggers** — list of trigger config items with type icon, type name, and trigger-specific summary text
5. **Oversight Rules** — list of rules showing condition (`--solar`, mono) and action
6. **Outputs** — type and target as field-value rows

**Actions footer:**
- "Open in Visualizer" button (secondary) — navigates to definition DAG view
- "Execute Now" button (primary, `--bio`) — fires the strategy via `POST /strategies/execute`

**Dependencies to carry over:**
- `@xyflow/react` ^12.0.0 — DAG rendering engine
- `dagre` ^0.8.5 — automatic graph layout

**Key adaptation:** The viz app currently fills the entire viewport (`width: 100vw; height: 100vh`). In the Narrative Flow frontend, the DAG canvas lives inside a content panel (below the nav bar, potentially alongside the slide-over detail panel). The canvas component must accept width/height as props or use a resize observer to fill its container.

**Routing:** The DAG visualizer is reached via:
- `/strategies/:id` — strategy detail page with embedded definition DAG
- `/strategies/:id/exec/:eid` — live execution view with real-time status overlaid on the DAG

**Definition-mode DAG (no execution):** When viewing a strategy that has never been executed, no DAG endpoint exists (the `/api/strategies/:id/dag` endpoint requires an execution ID). In this case, the frontend constructs the xyflow graph client-side from the `GET /api/strategies/definitions` response — specifically the `nodes` array with `depends_on` edges. Dagre layout is applied the same way as for execution DAGs. Node status indicators are omitted (all nodes shown as `pending`).

### Component 5: Live Execution View

When viewing a running execution (`/strategies/:id/exec/:eid`), the DAG canvas shows real-time node status.

**Data source:** `GET /strategies/:eid/status` polled every 2 seconds (matching the existing viz polling interval). Future enhancement: SSE stream for push-based updates.

**Node status rendering:**
- Pending: dim border, muted text
- Running: `--bio` glow pulse animation, spinner icon
- Completed: `--cyan` border, checkmark icon
- Failed: `--error` border, X icon
- Gate failed: `--solar` border, warning icon
- Suspended: `--nebular` border, pause icon

**Edge status rendering (via AnimatedEdge):**
- Pending: `--text-muted` dashed line
- Active (source completed, target running): `--bio` animated flow dots
- Completed (both endpoints done): `--cyan` solid line
- Failed: `--error` solid line

**Cost overlay:** Floating metric cards (top-right of DAG canvas) showing:
- Total cost (mono, large) with color thresholds: < $1 = `--text`, $1-5 = `--solar`, > $5 = `--error`
- Node progress: `{completed}/{total}` with optional `({failed} failed)` in `--solar`
- Duration (live-updating)
- Status badge

**Node click:** Opens a detail sub-panel showing:
- Node configuration (methodology, capabilities, inputs, outputs)
- Gate results (passed/failed, check expression, retry count)
- Cost and duration for completed nodes
- Error message for failed nodes
- Artifacts produced (expandable JSON view)

### Component 6: Execution History

A dedicated history view for a single strategy, accessible via:
- The "History" tab in the strategy detail slide-over
- The `/strategies/:id` page (below the DAG canvas)

**List rendering:** Each execution as a card showing:
- Execution ID (mono, truncated with tooltip for full ID)
- Start time and duration
- Status badge
- Cost (mono)
- Gate summary: `{passed}/{total} gates passed`, with individual gate results on hover
- Retro link (if `retro_path` is present): "View Retro" link navigating to the retrospective detail page
- "Compare" checkbox — selecting two executions enables the comparison view

**Filtering:**
- By status (completed, failed, running)
- By date range

**Sorting:**
- By start time (default, descending)
- By cost
- By duration

**Data source:** `GET /strategies` filtered by strategy ID.

### Component 7: Execution Comparison

Side-by-side view of two executions of the same strategy.

**Trigger:** Select two executions via checkboxes in the history list, then click "Compare Selected."

**Layout:** Two-column layout within the content area. Each column shows:

**Header row:** Execution ID, start time, duration, total cost, status badge.

**Node-by-node table:** One row per DAG node, columns:

| Node | Exec A Status | Exec A Duration | Exec A Cost | Exec B Status | Exec B Duration | Exec B Cost | Delta |
|------|--------------|----------------|-------------|--------------|----------------|-------------|-------|

- Duration deltas are color-coded: green if B is faster, red if B is slower
- Cost deltas are color-coded the same way
- Status differences are highlighted with background color

**Gate comparison table:** One row per gate, columns:
- Gate ID
- Exec A result (passed/failed + check expression result)
- Exec B result
- Highlight if result differs

**Summary cards:**
- Total cost delta (with percentage)
- Total duration delta (with percentage)
- Gate pass rate comparison

**Data source:** `GET /strategies/:eid/status` for each execution ID.

**Known constraint:** Execution comparison only works for executions still in the bridge's in-memory execution store, which is TTL-bounded (executions are evicted after the configured dead session TTL). Comparing historical executions that have been evicted from memory will return 404s. Persistent execution storage is out of scope for this PRD.

### Component 8: Gantt Timeline

A horizontal timeline visualization showing parallel execution timing.

**Layout:** Appears below the DAG canvas on the execution detail page. Full width of the content area.

**Rendering:**
- X-axis: time (starting at execution start, ending at completion)
- Y-axis: one row per node, ordered by topological sort level
- Each node is a horizontal bar:
  - Left edge = node start time
  - Right edge = node end time (or current time if running)
  - Color: `--bio` for completed, `--error` for failed, `--nebular` for suspended, animated gradient for running
  - Width encodes duration
- Critical path nodes highlighted with a stronger color (full opacity) and a connecting line

**Interaction:**
- Hover a bar: tooltip showing node ID, duration, cost, status
- Click a bar: scrolls the DAG canvas to center on that node and opens its detail panel

**Data source:** `node_results` from `GET /strategies/:eid/status`, which includes `duration_ms` per node. Node start times are derived from topological ordering and dependency completion timestamps.

**Backend prerequisite:** The Gantt timeline requires per-node `started_at` timestamps in the `node_results` response. Currently, `GET /strategies/:eid/status` returns `duration_ms` but not explicit start times. A backend delta is needed to record and return `started_at` for each node. Without this, the Gantt can only approximate start times from dependency completion order, which is imprecise for parallel nodes. This prerequisite is tracked in the Phase 3d deliverables.

**Implementation note:** This is a custom SVG or Canvas rendering — no external Gantt library dependency. The visualization is simple enough (rectangular bars on a timeline) that a purpose-built component is cleaner than adapting a general-purpose Gantt library to the Narrative Flow design language.

### Component 9: Execute Button

The "Execute Now" button in the strategy detail panel fires a strategy manually.

**Flow:**
1. Human clicks "Execute Now" in the slide-over panel
2. Confirmation dialog appears:
   - Strategy name and ID
   - Context inputs form: editable fields for each `context.inputs` entry, pre-filled with defaults
   - "Execute" (primary) and "Cancel" (secondary) buttons
3. On confirm: `POST /strategies/execute` with `{ strategy_path, context_inputs }`. The request body uses `strategy_path` from the definition (the file path within `.method/strategies/`), not the strategy ID.
4. On success: toast notification "Strategy {name} started" with link to live execution view
5. On error: toast notification with error message

**Context input form:** Each `context.inputs` entry from the strategy YAML becomes a form field:
- `type: string` renders as a text input
- `type: number` renders as a number input
- `type: object` renders as a JSON textarea
- Default values pre-fill the fields

**Why third:** The DAG visualizer is the most complex component. It depends on the strategy detail page (Phase 3b) for navigation context.

### Phase 3d: Execution History + Comparison + Gantt

**Deliverables:**
- Execution history list with filtering and sorting
- Comparison checkbox selection and side-by-side view

- Gantt timeline SVG component
- Critical path highlighting
- Retro links in history entries

**Why last:** History and comparison are analytical features that build on top of the execution infrastructure. The Gantt timeline is a new visualization that doesn't exist in the current viz app.

## 6. Narrative Flow Compliance

This section maps each component to the Narrative Flow design language principles.

### Attention First

- Currently running executions appear at the top of the strategy card grid with pulse-glow animation
- Failed executions use `--error` dots in the timeline — visually prominent
- The execute confirmation dialog prevents accidental fires

### Time as the Organizing Principle

- Execution history is a vertical timeline, chronologically ordered
- The Gantt timeline is explicitly temporal
- Strategy cards sort by last execution time

### Progressive Disclosure

| Tier | What the human sees | Time to process |
|------|-------------------|-----------------|
| Tier 1 (Glance) | Strategy card grid — name, status badge, mini-DAG, last execution time | 1 second |
| Tier 2 (Scan) | Execution timeline — cost, duration, gate summary per execution | 5 seconds |
| Tier 3 (Deep dive) | Slide-over detail panel, DAG visualizer, comparison view, Gantt | As needed |

### Color Semantics (per Design Language)

| Element | Color | Token |
|---------|-------|-------|
| Strategy ID text | Bioluminescent | `--bio` |
| Methodology nodes/badges | Nebular | `--nebular` |
| Script nodes/badges | Bioluminescent | `--bio` |
| Gate badges | Cyan | `--cyan` |
| Running execution | Bio glow pulse | `--bio-glow` |
| Completed execution | Cyan | `--cyan` |
| Failed execution | Error | `--error` |
| Oversight conditions | Solar Warmth | `--solar` |
| Trigger badges | Type-specific (manual = dim, file_watch = solar, git = nebular) | Various |

### Animation Compliance

| Animation | Spec | Follows DL? |
|-----------|------|-------------|
| Card entrance | 400ms ease-out, stagger 100ms | Yes (timeline item entrance) |
| Card hover lift | 200ms ease-out | Yes (card hover lift) |
| Running card pulse | 2500ms ease-in-out | Yes (badge pulse) |
| Slide-over open | 300ms ease-out | Yes (slide-over open) |
| Timeline item entrance | 400ms ease-out, stagger 100ms | Yes (timeline stagger) |
| Gantt bar fill | 800ms ease-out | Yes (progress bar fill) |
| All animations | Disabled on `prefers-reduced-motion` | Yes (required) |

## 7. Success Criteria

1. **Definition browsing works:** All strategies in `.method/strategies/` appear as cards with correct name, ID, mini-DAG thumbnail, trigger badges, and last execution summary. No raw YAML reading required.

2. **Detail panel renders structurally:** Clicking a strategy card opens a slide-over panel showing nodes, gates, triggers, and oversight rules as structured UI elements, not raw YAML.

3. **DAG visualizer is absorbed:** The xyflow DAG canvas renders within the Narrative Flow frontend at `/strategies/:id/exec/:eid`. All node types (methodology, script, gate) display correctly with Narrative Flow colors. The standalone `GET /viz/*` route is removed.

4. **Live execution updates:** Node status, cost, and gate results update in real-time (< 3 second latency) on the DAG canvas during an active execution.

5. **Execute button works:** Clicking "Execute Now" with valid context inputs fires a strategy and shows a toast with a link to the live execution view.

6. **Execution history is browsable:** Past executions are listed with status, cost, duration, and gate summaries. Filtering by status and sorting by time/cost/duration work correctly.

7. **Comparison works:** Selecting two executions shows a side-by-side view with node-by-node timing and cost deltas, color-coded for improvement/regression.

8. **Gantt timeline renders:** Parallel execution timing is visualized as horizontal bars with correct start/end times and critical path highlighting.

9. **No regression:** Existing `POST /strategies/execute`, `GET /strategies/:id/status`, and `GET /api/strategies/:id/dag` endpoints continue to work unchanged.

10. **Responsive:** Strategy cards stack to 1-column below 1200px. Slide-over becomes full-screen below 768px. DAG canvas is scrollable/zoomable at all breakpoints.

## 9. Out of Scope

- **Strategy YAML editor** — editing strategy definitions from the UI. Strategy files are code-reviewed artifacts; editing belongs in an IDE.
- **Strategy creation wizard** — generating new strategy YAML from a UI form. The YAML schema is documented in PRD 017; authoring is a developer task.
- **Execution replay** — scrubbing through a past execution's timeline to see node status at any point in time. Requires execution event log persistence not yet implemented.
- **Artifact deep inspector** — browsing and downloading individual artifacts produced by nodes. The detail panel shows artifact keys; full artifact content viewing is deferred.
- **Multi-strategy Gantt** — showing executions from different strategies on a single Gantt timeline. This PRD covers per-strategy timelines only.
- **Cost anomaly detection** — flagging executions that cost significantly more than historical average. Deferred to the Analytics page (PRD 019.x).
- **Strategy dependency graph** — visualizing which strategies trigger other strategies (via channel_event triggers). Deferred to the Triggers page (PRD 019.x).

## 10. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **xyflow bundle size** — adding @xyflow/react increases the frontend bundle by ~200KB | HIGH | LOW | Code-split the DAG visualizer as a lazy-loaded route. Users who never visit the strategy pages don't pay the cost. |
| **Dagre layout performance** — strategies with 15+ nodes may cause visible layout delay | LOW | LOW | Dagre layout is fast (< 50ms for 20 nodes). Memoize layout computation with `useMemo` keyed on DAG structure hash. |
| **YAML parsing on every request** — `GET /api/strategies/definitions` reparses all files per request | MEDIUM | LOW | Acceptable for < 20 files. Add mtime-based caching if the directory grows beyond 50 strategies. |
| **Comparison data volume** — two full execution status responses fetched simultaneously | LOW | LOW | Execution status responses are small (< 10KB each). No pagination needed. |
| **Gantt timeline for long executions** — executions lasting 30+ minutes produce a very wide timeline | MEDIUM | LOW | Horizontal scrolling with a minimap scrubber. Time axis auto-scales to fit the viewport width with scroll for detail. |
| **Viz absorption regressions** — ported components may break due to styling or import changes | MEDIUM | MEDIUM | Port components incrementally. Keep `GET /viz/*` operational until all features are verified in the new frontend. Deprecation is a separate commit. |
