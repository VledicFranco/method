## 1. Purpose and Problem Statement

### The registry is invisible to humans

The method registry (`registry/`) contains all compiled methodologies, methods, protocols, their domain theories, step DAGs, compilation records, known WIP items, and evolution history. This is the intellectual backbone of the system — every methodology routing decision, every step the agent follows, every axiom it enforces originates from registry YAML files. But the registry has zero human-facing UI. A human who wants to understand what methods exist, what compilation state they are in, or what domain theory drives a particular method must open YAML files in an editor and mentally parse nested structures.

### Methodology users need a reference browser

When an orchestrator commissions work or a council session reviews methodology routing, the human needs to quickly look up: What steps does M1-IMPL have? What are the axioms of M1-COUNCIL? Is P3-GOV fully compiled or still draft? What WIP items remain on M6-ARFN? What protocols are installed and are they promoted or still in trial? Today these questions require navigating a directory tree and reading raw YAML. The Registry UI makes this a 5-second scan or one-click deep dive.

### What this PRD delivers

A **Registry UI** embedded in the bridge frontend that provides:
- A tree view of all methodologies and their methods with status indicators
- A tabbed detail view for any method (navigation, domain theory, steps, compilation record, known WIP)
- Protocol lifecycle display with promotion criteria progress
- A manifest view showing installed methodologies with version sync status
- Four new bridge API endpoints (three read endpoints that serve parsed registry data as JSON, plus a cache invalidation endpoint)

**Architectural constraint (DR-03/DR-04):** The API endpoints live in `@method/bridge`. They read YAML files from the registry directory using `js-yaml` (DR-05) and serve parsed JSON. The frontend consumes these endpoints. No registry-specific logic is added to `@method/core` — the registry is a file-based artifact store, not a runtime concern.

## 2. Components

### Component 1: Registry API Endpoints

Four HTTP endpoints on the bridge server. Three read endpoints parse registry YAML files on demand and return JSON. One write endpoint invalidates the cache.

#### `GET /api/registry`

Returns the full registry tree structure. Scans the `registry/` directory, reads each methodology-level YAML (e.g., `P1-EXEC.yaml`, `P2-SD.yaml`), and each method-level YAML within subdirectories. Returns a tree:

```json
{
  "methodologies": [
    {
      "id": "P0-META",
      "name": "Meta-Methodology",
      "version": "1.0",
      "status": "compiled",
      "method_count": 9,
      "methods": [
        {
          "id": "M1-MDES",
          "name": "Method Design",
          "version": "1.0",
          "status": "compiled",
          "type": "method",
          "wip_count": 0
        },
        {
          "id": "M2-MDIS",
          "name": "Method Discovery",
          "version": "1.0",
          "status": "compiled",
          "type": "method",
          "wip_count": 0
        },
        {
          "id": "RETRO-PROTO",
          "name": "Retrospective Protocol",
          "version": "1.0",
          "status": "promoted",
          "type": "protocol",
          "wip_count": 0
        },
        {
          "id": "CMEM-PROTO",
          "name": "Collective Memory Protocol",
          "version": "1.0",
          "status": "promoted",
          "type": "protocol",
          "wip_count": 0
        }
      ]
    },
    {
      "id": "P3-GOV",
      "name": "Governance Methodology",
      "version": "0.1",
      "status": "draft",
      "method_count": 4,
      "methods": []
    }
  ],
  "totals": {
    "methodologies": 6,
    "methods": 28,
    "protocols": 3,
    "compiled": 25,
    "draft": 6
  }
}
```

**Important:** The example JSON above is illustrative. The API must scan actual YAML files in the registry directory and return real data. Mock data or hardcoded examples must never be returned by the endpoint.

**Caching:** The response is cached in memory with a TTL of 60 seconds. Registry files change rarely (only during method compilation), so aggressive caching is appropriate. The cache is invalidated on `POST /api/registry/reload`.

**Directory scanning logic:**
1. Read top-level directories in `registry/` — each directory whose name matches `P[0-9]-*` or `P-*` is a methodology (skip `instances/`, `submissions/`).
2. For each methodology directory, read the methodology-level YAML (`{dir}/{id}.yaml`) for metadata.
3. For each subdirectory within the methodology directory (matching `M[0-9]-*` or `*-PROTO`), read the method-level YAML for metadata.
4. **Protocol file resolution:** Scan the methodology directory for all `*.yaml` files, parse the YAML, and extract the `protocol.id` or `method.id` field. Match by parsed ID, not filename convention. Protocol filenames are not standardized (e.g., `RETROSPECTIVE-PROTOCOL.yaml`, `STEERING-PROTOCOL.yaml`, `CMEM-PROTOCOL.yaml`) — the parser must rely on the YAML content to determine whether a file is a method or protocol and what its canonical ID is. Files that contain a top-level `protocol` key are included as type `protocol` under the parent methodology.

#### `GET /api/registry/:methodology/:method`

Returns the full parsed YAML for a specific method, converted to JSON. The response preserves the complete structure: `method` (or `protocol`), `navigation`, `domain_theory`, `steps` (or `step_dag`), `roles`, `compilation_record`, `known_wip`, `success_profile`, and any other top-level keys.

Path parameters:
- `:methodology` — methodology ID (e.g., `P1-EXEC`)
- `:method` — method ID (e.g., `M1-COUNCIL`) or protocol ID (e.g., `STEER-PROTO`)

File resolution:
1. Method: `registry/{methodology}/{method}/{method}.yaml`
2. Protocol (methodology-level): `registry/{methodology}/{protocol-filename}.yaml` — filename inferred from protocol ID mapping

Returns 404 if the file does not exist. Returns 422 if the YAML fails to parse (with the parse error message in the response body).

#### `GET /api/registry/manifest`

Returns the parsed `.method/manifest.yaml` augmented with sync status. For each installed methodology or protocol, compares the manifest version against the current registry version and reports staleness:

```json
{
  "project": "pv-method",
  "last_updated": "2026-03-16",
  "installed": [
    {
      "id": "P2-SD",
      "type": "methodology",
      "version": "2.0",
      "registry_version": "2.0",
      "sync_status": "current",
      "card": "project-card.yaml",
      "instance_id": "I2-METHOD",
      "artifacts": ["project-card.yaml", "CHANGELOG.yaml", "..."]
    },
    {
      "id": "STEER-PROTO",
      "type": "protocol",
      "version": "0.1",
      "registry_version": "0.1",
      "sync_status": "current",
      "status": "draft",
      "extends": "P1-EXEC",
      "artifacts": ["council/TEAM.yaml", "..."]
    }
  ]
}
```

**Note on `extends` field:** Extension protocols (protocols that extend an existing methodology, e.g., STEER-PROTO extends P1-EXEC) include an `extends` field in the manifest YAML. The API preserves this field in the response so the frontend can indicate the parent methodology relationship.

**Note on auxiliary files:** Methodology directories may contain auxiliary YAML files such as `CHANGELOG.yaml`, `EVOLUTION-LEDGER.yaml`, `KNOWN-GAPS.yaml`, and similar non-method/non-protocol files. The scanner ignores these except for `*-PROMOTION.yaml` files, which are needed by the protocol lifecycle view (Component 4) to display promotion criteria and evidence.

Sync status values:
- `current` — manifest version matches registry version
- `outdated` — registry has a newer version than the manifest
- `ahead` — manifest version is ahead of registry (shouldn't happen, flagged as warning)
- `not_found` — methodology/protocol not found in registry

#### `POST /api/registry/reload`

Invalidates the in-memory registry cache and forces a fresh scan of the `registry/` directory on the next `GET /api/registry` request.

**Response:**

```json
{
  "status": "ok",
  "message": "Registry cache invalidated. Next request will re-scan."
}
```

This is useful after method compilation or manual edits to registry YAML files, when the human wants immediate visibility of changes without waiting for the cache TTL to expire.

### Component 3: Method Detail View (Main Content)

The right panel (remaining width, max-width 720px) shows full detail for the selected method. Organized as a tabbed interface with 5 tabs.

#### Header

Displays above the tabs for every method:
- **Method ID** — mono font, `--bio` color (e.g., `P1-EXEC / M1-COUNCIL`)
- **Method name** — display font, `--text-xl` (e.g., "Steering Council")
- **Status badge** — compiled/draft/promoted, color-coded per design language
- **Version** — mono font
- **Compilation date** — dim text

#### Tab 1: Navigation

Renders the method's `navigation` section as a card grid (2-column, with `why` spanning full width):

| Card | Content |
|------|---------|
| **What** | `navigation.what` |
| **Who** | `navigation.who` |
| **Why** (full width) | `navigation.why` |
| **How** | `navigation.how` |
| **When** | `navigation.when_to_use` (bullet list) |

Cards use `--abyss` background, `--border` border, 12px radius. Labels are mono uppercase in `--text-muted`.

If `navigation.when_not_to_use` exists, render a secondary "When Not To Use" card below the grid with `--error-dim` left border accent.

#### Tab 2: Domain Theory

Renders the method's `domain_theory` section in three subsections:

**Sorts table:**
| Column | Source |
|--------|--------|
| Sort | `sorts[].name` |
| Description | `sorts[].description` |
| Cardinality | `sorts[].cardinality` |

**Predicates table:**
| Column | Source |
|--------|--------|
| Predicate | `predicates[].name` |
| Signature | `predicates[].signature` (mono font) |
| Description | `predicates[].description` |

**Function Symbols table** (if present):
| Column | Source |
|--------|--------|
| Symbol | `function_symbols[].name` |
| Signature | `function_symbols[].signature` (mono font) |
| Totality | `function_symbols[].totality` |
| Description | `function_symbols[].description` |

**Axioms list:**
| Column | Source |
|--------|--------|
| ID | `axioms[].id` (mono font) |
| Name | `axioms[].name` |
| Statement | `axioms[].statement` (mono font, smaller size) |
| Rationale | `axioms[].rationale` (dim, expandable if long) |

Tables follow the `data-table` pattern from the design language: mono headers, `--border` separators, `--text-dim` body text.

#### Tab 3: Steps

Two sections:

**Step DAG visualization:**
Renders the method's step sequence as a horizontal flow diagram. Each step is a circle node (48px, `--abyss` background, `--bio` border) with the step symbol inside (e.g., `s_0`). Nodes are connected by directional arrows (2px `--border` lines with arrowheads). Below each node, a label shows the step name.

For methods with linear step sequences (most methods), use the simple horizontal layout from the mock. For methods with branching or re-entry edges (e.g., M1-IMPL has a re-entry from sigma_B4 back to sigma_A1), use xyflow for a proper DAG layout within this tab. The decision to use xyflow vs simple SVG is per-method based on the `composability_claims` — if any claim has `type: re_entry` or `type: conditional`, use xyflow.

On step node hover: show `--bio-glow` box shadow. On step node click: scroll to that step's row in the detail table below and highlight it.

**Step detail table:**
| Column | Source |
|--------|--------|
| Step | Step symbol (e.g., `sigma_0`) |
| Name | Step name |
| Precondition | `precondition` or `initial_condition_claim` |
| Postcondition | `postcondition` or `terminal_condition_claim` |
| Execution Binding | If declared per step |
| Guidance Adequacy | `guidance_adequacy` result (ADEQUATE/INADEQUATE) |

#### Tab 4: Compilation Record

Renders `compilation_record.gates` as a checklist. Each gate is a card (from mock: `gate-item` pattern):

- **Status circle** — 28px, color-coded:
  - PASS: `--bio-dim` background, `--bio` checkmark
  - FAIL: `--error-dim` background, `--error` X mark
  - DEFERRED: `--text-muted` background, dash
  - PASS_WITH_WIP: `--solar-dim` background, `--solar` checkmark with exclamation
- **Gate name** — mono font (e.g., "G0 — YAML Structure")
- **Note** — dim text, the `note` field from the compilation record

The 7 gates in order:
| Gate | Name |
|------|------|
| G0 | YAML Structure |
| G1 | Navigation Completeness |
| G2 | Domain Theory |
| G3 | Step DAG |
| G4 | Predicate Coverage |
| G5 | Axiom Grounding |
| G6 | Cross-Reference |

If the method has no `compilation_record` (e.g., protocol files), show a notice: "No compilation record. Protocols follow a lifecycle model instead of compilation gates."

#### Tab 5: Known WIP

Renders `known_wip` items as cards (from mock: `wip-item` pattern):

- **Status badge** — `open` (solar) or `resolved` (cyan)
- **WIP ID and name** — mono font
- **Description** — dim text

If no WIP items exist, show: "No known work-in-progress items."

### Component 4: Protocol Lifecycle View

When a protocol is selected in the tree (e.g., RETRO-PROTO, STEER-PROTO), the detail view adapts:

**Header:** Shows protocol name, version, and lifecycle status badge:
- `draft` — gray badge (`--text-dim`)
- `trial` — solar badge (`--solar`) with flask/test-tube icon
- `promoted` — cyan badge (`--cyan`) with star icon

**Lifecycle pipeline:** A horizontal progress indicator showing the stages: `draft -> trial -> evaluate -> promote`. The current stage is highlighted with the appropriate color. Completed stages show checkmarks.

**Promotion criteria** (if available from a promotion record YAML like `RETRO-PROTO-PROMOTION.yaml`):
A checklist of criteria with measured values:

| Criterion | Target | Measured | Status |
|-----------|--------|----------|--------|
| Retrospective count | >= 10 | 11 | PASS |
| Gap candidates surfaced | >= 2 | 4 | PASS |
| Evolution triggered | >= 1 | 2 | PASS |
| Evolution validated | >= 1 | 1 | PASS |
| Self-referential test | >= 1 | 1 | PASS |

**Installation details:** From `navigation` and `installation` fields of the protocol YAML — what artifacts are required, which projects have installed it.

**Tabs:** Protocols reuse the same tab structure where applicable:
- **Navigation** — same as methods
- **Domain Theory** — if present
- **Installation** — artifact requirements and template notes
- **Promotion** — criteria, evidence, status
- **Known WIP** — if present

## 3. Implementation Order

### Phase 1: API + Tree + Method Detail (Foundation)

**Deliverables:**
- `GET /api/registry` endpoint with directory scanning and caching
- `GET /api/registry/:methodology/:method` endpoint with YAML parsing
- `GET /api/registry/manifest` endpoint with version sync detection
- `POST /api/registry/reload` endpoint for cache invalidation
- Registry tree sidebar component (collapsible, searchable)
- Method detail view: Navigation tab, Domain Theory tab, Steps tab (simple linear layout only), Compilation Record tab, Known WIP tab
- Route: `/registry` (tree + detail)

**Why first:** The tree and detail view are the core UC-68/UC-69/UC-70 experience. The API endpoints are the data foundation everything else builds on.

**Estimated scope:** ~800 lines backend (3 endpoints, YAML scanning, caching), ~1200 lines frontend (tree component, 5 tab components, data fetching).

### Phase 2: Protocol Lifecycle + Manifest + DAG Enhancement

**Deliverables:**
- Protocol lifecycle view with promotion criteria display
- Manifest view with sync status badges and outdated warnings
- Step DAG with xyflow for methods that have branching/re-entry edges
- Evolution History tab (placeholder or initial implementation if EVOLUTION-LEDGER parsing is straightforward)

**Why second:** Protocol lifecycle (UC-71) and manifest (UC-72) are secondary use cases that build on the Phase 1 data infrastructure. The xyflow integration for complex step DAGs requires the simple version to be working first.

**Estimated scope:** ~400 lines backend (promotion record parsing, evolution ledger parsing), ~800 lines frontend (protocol components, manifest view, xyflow integration).

## 4. Success Criteria

1. **Tree loads correctly:** The registry tree displays all methodologies and methods found in the `registry/` directory. Methodology count, method count, and status icons match the actual YAML files.

2. **Method detail renders:** Selecting any method in the tree loads and displays its full detail — all 5 tabs populated from the parsed YAML. No raw YAML is shown; everything is rendered as structured UI components.

3. **Domain theory is human-readable:** Sorts, predicates, function symbols, and axioms render as clean tables. Formal notation (sort signatures, predicate signatures) uses mono font. Descriptions are in body font. A non-developer can read the domain theory and understand what the method is about.

4. **Compilation gates are a checklist:** Each gate shows PASS/FAIL/WIP with a colored indicator and note. The human can glance at the checklist and know the compilation state of any method in under 3 seconds.

5. **Search filters the tree:** Typing "IMPL" in the search box filters the tree to show only methods containing "IMPL" in their ID or name, with parent methodologies expanded. Clearing the search restores the full tree.

6. **Protocol lifecycle is visible:** Selecting a protocol shows its lifecycle stage (draft/trial/promoted) with a visual progress indicator. If promotion criteria data exists, it renders as a criteria checklist with measured values.

7. **Manifest sync status works:** The manifest view shows each installed methodology and protocol with its version compared against the registry. `outdated` status produces a visible warning banner.

8. **API returns valid JSON:** All four endpoints return well-formed JSON. Invalid YAML files produce 422 responses with informative error messages rather than 500 crashes. The `POST /api/registry/reload` endpoint returns a confirmation response.

9. **Performance:** The `/api/registry` endpoint responds in under 500ms on first call (cold cache) with the full registry (~30 YAML files). Subsequent calls return from cache in under 10ms.

10. **No regression:** Existing bridge endpoints, dashboard, and strategy visualizer continue to work unchanged. The new endpoints are additive.

## 6. Out of Scope

- **Registry editing.** The UI is read-only. Modifying registry YAML files requires direct file editing or the method compilation process (M1-MDES, M3-MEVO). A WYSIWYG registry editor would violate DR-01/DR-02 (registry files are production artifacts with compilation status).

- **Method execution from registry.** The registry browser does not provide a "run this method" button. Method execution is handled by the methodology runtime (MCP tools) and commissioned through the council workflow.

- **Cross-project registry browsing.** This PRD serves a single project's registry. Multi-project registry comparison (e.g., comparing I1-T1X's installed methods vs I2-METHOD's) is a future capability that would require the inter-project council communications system.

- **Full-text search across YAML content.** The tree search filters by method ID and name only. Searching for text inside axiom statements, step guidance, or predicates is deferred. If needed, it would require a server-side search index.

- **YAML syntax highlighting.** The UI renders structured components, not raw YAML. A "view raw YAML" toggle is not included in Phase 1 — the structured rendering is the primary value. If users request raw YAML access, it can be added as a minor enhancement.

- **Evolution history visualization.** The EVOLUTION-LEDGER.yaml analysis (gap candidate tracking, pre/post observation rates, version timelines) is deferred to Phase 2 as noted in Component 6.

- **Real-time registry change detection.** The API uses TTL-based caching. File-watch-based cache invalidation (detecting when a registry YAML file changes and pushing updates to the frontend) is unnecessary given the low change frequency of registry files.

## 7. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Registry YAML schema variance** — different methods have different YAML structures (methods vs protocols vs methodology-level files) | HIGH | MEDIUM | The API normalizes all files into a common JSON envelope with a `type` field (`method`, `protocol`, `methodology`). The frontend dispatches to type-specific renderers. Missing fields render as "N/A" or are hidden, not errored. |
| **Large YAML files slow parsing** — some method YAMLs (e.g., M1-COUNCIL, M1-IMPL) are 500+ lines | LOW | LOW | `js-yaml` parses 1000-line YAML in <5ms. The `/api/registry` tree endpoint reads only top-level metadata, not full files. Full parsing only happens for `/api/registry/:methodology/:method` which loads one file at a time. |
| **Protocol promotion data not standardized** — promotion criteria records exist for RETRO-PROTO but the format may not be consistent for future protocols | MEDIUM | LOW | The protocol lifecycle view renders what exists. If no promotion record file is found, show "No promotion criteria recorded" instead of erroring. Define a lightweight schema for promotion records that future protocols should follow. |
| **Step DAG complexity** — some methods have complex step graphs with conditional edges, re-entry, and parallel branches | MEDIUM | MEDIUM | Phase 1 renders linear step sequences as simple horizontal flows (covers ~80% of methods). Phase 2 adds xyflow for complex DAGs. Methods with complex steps that hit Phase 1 show a flat ordered list instead of a graph. |
| **Stale cache after method compilation** — a human compiles a method but the API returns the old cached version | MEDIUM | LOW | 60-second cache TTL means the delay is at most 1 minute. `POST /api/registry/reload` endpoint allows manual cache invalidation. The frontend shows cache age in a subtle timestamp: "Registry data from 45s ago." |
| **Frontend bundle size** — xyflow adds ~180KB gzipped for the DAG rendering | LOW | LOW | xyflow is already bundled in the strategy visualizer. Code-split the registry page to load xyflow lazily only when a complex DAG method is selected. Simple step sequences use plain SVG with zero extra dependencies. |

### Relationship to Narrative Flow Design Language

The Registry UI follows all Narrative Flow principles but with a notable adaptation: **the timeline is not the organizing principle here.** Registry data is structural, not temporal. The organizing principle is the hierarchy: methodology > method > detail tabs. The design language's progressive disclosure tiers apply:

- **Tier 1 (Glance):** Tree with status icons — 1 second to see which methods are compiled/draft/WIP.
- **Tier 2 (Scan):** Method header + tab overview — 5 seconds to understand what a method does and its compilation state.
- **Tier 3 (Deep dive):** Domain theory, step DAG, individual axiom rationale — as long as needed.

Color system, typography, spacing, animation, and component patterns all follow the design language reference exactly.
