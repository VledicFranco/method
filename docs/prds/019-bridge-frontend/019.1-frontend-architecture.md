## 1. Purpose and Problem Statement

### Four disconnected UIs with no shared navigation

The bridge currently serves four separate UI experiences: a server-rendered HTML dashboard (`GET /dashboard`), a per-session live terminal page (`GET /sessions/:id/live`), a per-session transcript browser (`GET /sessions/:id/transcript`), and a React SPA strategy visualizer (`GET /viz/*`). These share no navigation, no state, no design system, and no cross-linking. A human monitoring agent work must maintain mental context across browser tabs with no way to navigate from a channel event to the session that produced it, or from a strategy execution to the agents running inside it.

### No foundation for the features the system needs

The bridge frontend redesign plan (84 use cases across 10 feature areas) requires: real-time data subscriptions, component reuse across pages, global state for attention items and notifications, client-side routing for deep linking, and a design system implementing the Narrative Flow design language. None of this is achievable by extending the current server-rendered HTML dashboard or by adding more isolated pages. A unified SPA architecture is the prerequisite for every subsequent UI PRD.

### What this PRD delivers

The **frontend architecture** — the React SPA foundation that all other bridge UI PRDs build on. It defines the tech stack, project structure, component library hierarchy, data layer (API client, server state, UI state, real-time subscriptions), routing scheme, build and deployment pipeline, design system integration with Vidtecci tokens, testing strategy, and the migration path from the existing UI. This PRD does NOT implement any pages — it delivers the skeleton, the design system primitives, and the data plumbing. Subsequent PRDs (019.2 through 019.N) implement individual feature areas on this foundation.

## 2. Components

### Component 1: Tech Stack

| Layer | Choice | Rationale |
|-------|--------|-----------|
| **Framework** | React 18+ with TypeScript | Existing viz app already uses React 18. Large ecosystem for the component types needed (terminal emulation, DAG visualization, charts). TypeScript required by project convention. |
| **Build** | Vite 6 | Existing viz app already uses Vite 6. Fast HMR for development. Native ESM output. Efficient code splitting via Rollup. |
| **Routing** | React Router v7 | Mature, well-documented, supports lazy route loading for code splitting. Data loaders and actions for route-level data prefetching. The viz app has no routing — this is new. TanStack Router considered but React Router has wider community knowledge and simpler migration path. |
| **Server state** | TanStack Query v5 | Caching, deduplication, background refetch, and stale-while-revalidate for all API calls. Eliminates the need for manual fetch-and-setState patterns. Integrates with SSE subscriptions for real-time invalidation. |
| **UI state** | Zustand v5 | Lightweight store for UI-only state (selected session, panel open/closed, view preferences, notification settings, attention item acknowledgments). No boilerplate — simpler than Redux for the state shapes needed. |
| **Styling** | Tailwind CSS v4 | Utility-first CSS with design token integration via CSS custom properties and `theme()` config. Narrative Flow tokens map directly to Tailwind theme extensions. JIT compilation produces minimal CSS. |
| **Component variants** | class-variance-authority (cva) | Type-safe variant definitions for component styling (e.g., `Badge` with `status` variant). Integrates cleanly with Tailwind classes. |
| **Charts** | Recharts | Declarative React chart library with composable components. Sufficient for sparklines, area charts, bar charts, treemaps, and time-series. Smaller bundle than visx for the chart types needed. |
| **Terminal** | xterm.js + @xterm/addon-fit | Existing bridge SSE stream (`/sessions/:id/stream`) already produces raw PTY output. xterm.js is already proven in the live output page. The addon-fit handles responsive terminal sizing. |
| **DAG visualization** | @xyflow/react v12 | Existing viz app uses xyflow for strategy DAG rendering. Absorb the existing custom nodes and layout logic. |
| **Layout** | dagre | Existing viz app uses dagre for automatic DAG layout. Retain as a dependency. |
| **Icons** | Lucide React | MIT-licensed, tree-shakeable icon library. Consistent stroke style that aligns with the Narrative Flow design language's organic geometry principle. |
| **React plugin** | @vitejs/plugin-react | Required dev dependency for Vite to handle JSX transform and React Fast Refresh during development. |
| **Fonts** | Space Grotesk + Inter + JetBrains Mono | Specified by the design language: Space Grotesk for headings (geometric precision with personality), Inter for body (maximum legibility), JetBrains Mono for technical data (IDs, costs, tokens). Loaded from Google Fonts with `font-display: swap`. For air-gapped environments, self-host fonts in `public/fonts/` with local `@font-face` declarations in `fonts.css` — Google Fonts CDN is the default. |

**Why `packages/bridge/frontend/` and not a separate package:** The frontend is a pure consumer of the bridge HTTP API. It has no code dependencies on `@method/bridge` TypeScript modules. But it is conceptually part of the bridge product — it shares the same deployment unit (bridge serves its static assets), the same versioning, and the same PRD lineage. Keeping it in `packages/bridge/frontend/` avoids a new workspace entry while maintaining clear separation (own `package.json`, own `tsconfig.json`, own build).

**Why not extend `packages/bridge/viz/`:** The viz app is a 4-file, single-concern SPA (DAG rendering). Expanding it to a full multi-page application with routing, state management, a design system, and 13 pages would require rewriting its structure entirely. It is simpler to create a clean foundation and absorb the viz's DAG components (`StrategyNode.tsx`, `GateNode.tsx`, layout logic) into the new app's `components/domain/` layer.

### Component 3: Mount Path and Backward Compatibility

| Path | Serves | Change |
|------|--------|--------|
| `/app/*` | New React SPA (client-side routing) | **New.** Bridge serves `index.html` for all `/app/*` routes (SPA fallback). |
| `/dashboard` | Existing server-rendered HTML dashboard | **No change.** Remains as-is during migration. |
| `/viz/*` | Existing strategy visualizer SPA | **No change.** Deprecated once DAG views are absorbed into `/app/strategies`. |
| `/sessions/:id/live` | Existing live terminal page | **No change.** Deprecated once terminal embed works in `/app/sessions/:id`. |
| `/sessions/:id/transcript` | Existing transcript browser page | **No change.** Deprecated once transcript views are in `/app/sessions/:id`. |
| All API endpoints | JSON API | **No change.** The frontend is a pure consumer. |

**SPA fallback implementation:** The bridge Fastify server adds a scoped `app.get('/app/*')` catch-all route that serves `packages/bridge/dist/app/index.html` (matching the existing `GET /viz/*` pattern in `strategy-viz-route.ts`). This enables React Router to handle client-side route resolution. Static assets (`/app/assets/*`) are served directly from the Vite build output directory. No `@fastify/static` dependency is needed — file serving uses manual `readFileSync` + MIME type dispatch, consistent with the viz route implementation.

```typescript
// In bridge server setup (packages/bridge/src/index.ts)
import { existsSync, readFileSync } from 'node:fs';
import { join } from 'node:path';

const MIME_TYPES: Record<string, string> = {
  html: 'text/html; charset=utf-8',
  js: 'application/javascript',
  css: 'text/css',
  svg: 'image/svg+xml',
  json: 'application/json',
  png: 'image/png',
  ico: 'image/x-icon',
  woff: 'font/woff',
  woff2: 'font/woff2',
  ttf: 'font/ttf',
};

const appDir = join(__dirname, 'app');

app.get('/app', async (_request, reply) => {
  return reply.redirect('/app/');
});

app.get('/app/*', async (request, reply) => {
  const rawPath = (request.params as Record<string, string>)['*'] || 'index.html';

  // Security: reject path traversal
  if (rawPath.includes('..')) {
    return reply.status(400).send({ error: 'Invalid path' });
  }

  const filePath = join(appDir, rawPath);

  // Try to serve the exact file
  if (existsSync(filePath)) {
    try {
      const content = readFileSync(filePath);
      const ext = rawPath.split('.').pop() ?? 'html';
      const mime = MIME_TYPES[ext] ?? 'application/octet-stream';
      return reply.type(mime).send(content);
    } catch {
      // Fall through to SPA fallback
    }
  }

  // SPA fallback: serve index.html for client-side routing
  const indexPath = join(appDir, 'index.html');
  if (existsSync(indexPath)) {
    return reply.type('text/html; charset=utf-8').send(readFileSync(indexPath));
  }

  return reply.status(404).send({
    error: 'Frontend not built. Run: npm run build:frontend',
  });
});
```

### Component 4: Component Library Architecture

Four-tier hierarchy with strict layering rules:

**Tier 1 — Primitives (`components/ui/`):**
Atomic UI elements with no domain knowledge. Accept Vidtecci theme tokens via Tailwind classes. Support all interaction states (hover, focus, active, disabled). Every primitive exports a TypeScript interface for its props. Components use `cva` for variant definitions.

| Component | Variants | Key Props |
|-----------|----------|-----------|
| `Button` | `primary`, `secondary`, `ghost`, `danger` | `size`, `loading`, `disabled`, `leftIcon`, `rightIcon` |
| `Badge` | `status` (running/completed/failed/pending/suspended/warning) | `size`, `icon`, `label` |
| `Card` | `default`, `interactive`, `active` | `padding`, `onClick`, `selected` |
| `Input` | `default`, `search`, `error` | `label`, `placeholder`, `error`, `leftIcon` |
| `Select` | `default` | `options`, `value`, `onChange`, `placeholder` |
| `Tabs` | `default`, `pills` | `tabs`, `activeTab`, `onTabChange` |
| `Tooltip` | `default` | `content`, `position`, `delay` |
| `Modal` | `default`, `danger` | `open`, `onClose`, `title`, `children` |
| `Drawer` | `right`, `left` | `open`, `onClose`, `width`, `backdrop` |

**Tier 2 — Layout (`components/layout/`):**
Page structure components that compose primitives into spatial arrangements. No data fetching — they receive content as children or render props.

| Component | Purpose | Key Props |
|-----------|---------|-----------|
| `NavBar` | Sticky top navigation. Logo left, nav items center, actions right. Bioluminescent underline on active item. Badge counts for pending actions. | `items`, `activeItem`, `badges`, `onSearch` |
| `PageShell` | Wraps every page. Renders NavBar, content area (820px centered, or 1200px when `wide` is set), and optional slide-over panel region. | `title`, `breadcrumbs`, `children`, `wide` (expands max-width to 1200px for DAG and analytics views) |
| `AttentionBanner` | Full-width card below nav for human-in-the-loop items. Solar warmth left border. Collapsible. Items with action buttons. | `items`, `onAction`, `onDismiss` |
| `SlideOverPanel` | 420px right-side panel with backdrop blur. Tabbed content. Escape/backdrop close. Slide-in animation. | `open`, `onClose`, `title`, `tabs`, `activeTab` |
| `Breadcrumb` | Hierarchical navigation trail. | `segments` |

**Tier 3 — Data display (`components/data/`):**
Components that render typed data with semantic meaning. Accept data props and formatting options. No data fetching — hooks provide the data, these components render it.

| Component | Purpose | Key Props |
|-----------|---------|-----------|
| `MetricCard` | Label above, value below (mono font). Optional trend indicator and sparkline. Counter animation on load. | `label`, `value`, `trend`, `sparkline`, `format` |
| `Timeline` | Vertical timeline container. Left-side line with colored dots per event type. Time period grouping. Staggered entrance animation. | `events`, `onEventClick`, `loading` |
| `TimelineEvent` | Individual timeline event card. Title, context, action links. Semantic color dot. | `event`, `onClick`, `actions` |
| `StatusBadge` | Color-coded pill. Status icon + label. Maps status values to design language colors. | `status`, `size` |
| `ProgressBar` | 4px height, bio-colored fill. Pulse animation on running state. | `value`, `max`, `status`, `animate` |
| `Sparkline` | Inline micro-chart. SVG line with fill gradient. Draw animation on load. | `data`, `width`, `height`, `color` |
| `Chart` | Wrapper around Recharts for consistent styling. Applies Vidtecci tokens to axes, labels, and series colors. | `type`, `data`, `config` |

**Tier 4 — Domain (`components/domain/`):**
Components that embed domain knowledge (session shapes, strategy DAG semantics, trigger types). May compose multiple data display components. Still no data fetching — they receive domain objects as props.

| Component | Purpose | Composes |
|-----------|---------|----------|
| `SessionCard` | Session summary card. Nickname, status, methodology step, duration, cost, progress bar. Pulse glow when running. | `Card`, `StatusBadge`, `ProgressBar` |
| `StrategyCard` | Strategy summary. Name, execution status, node progress, cost, mini-DAG thumbnail. | `Card`, `StatusBadge`, `ProgressBar` |
| `TriggerCard` | Trigger summary. Type icon, strategy, enabled/disabled, fire rate sparkline, last fired. | `Card`, `StatusBadge`, `Sparkline` |
| `RetroCard` | Retrospective summary. Date, method, observation count, highest severity, improvement targets. | `Card`, `Badge` |
| `RFCCard` | RFC summary. Title, phase badge, author, revision count, review coverage. | `Card`, `Badge` |
| `RegistryTree` | Tree view of methodologies and methods. Expandable nodes with compilation status badges. | `Badge` |
| `TerminalEmbed` | xterm.js terminal embedded as a resizable panel. Connects to bridge SSE stream. Handles fit-on-resize. | xterm.js, `@xterm/addon-fit` |
| `CommissionBuilder` | Commission prompt editor with 7 expandable sections. Fire button spawns bridge session. | `Card`, `Button`, `Tabs` |

**Layering rule:** Tier N components may import from Tier N-1 or lower. Never upward. Domain components compose data components. Data components compose primitives. Layout components compose primitives. No circular dependencies.

**Tree-shakeability:** Every component is a named export from its own file. No barrel `index.ts` re-exports at the `components/` root. Import paths are explicit: `import { Badge } from '@/components/ui/Badge'`.

**Accessibility:** All interactive components support keyboard navigation. All components respect `prefers-reduced-motion` by conditionally applying animations. Color is never the sole indicator of status — icons and text labels always accompany color. Focus rings use `--bio` with `2px offset`.

### Component 5: Data Layer

Three concerns with strict separation:

**5a. API Client (`lib/api.ts`)**

A thin wrapper around `fetch()` that provides:
- Base URL configuration (defaults to `window.location.origin`, overridable via env var for development)
- JSON serialization/deserialization
- Error normalization (HTTP errors become typed `ApiError` objects)
- Request abort via `AbortSignal` for component unmount cleanup

```typescript
// Usage pattern
const api = createApiClient({ baseUrl: 'http://localhost:3456' });
const sessions = await api.get<SessionListResponse>('/sessions');
const result = await api.post<SpawnResponse>('/sessions', { workdir, nickname });
```

No auth headers — the bridge has no authentication (out of scope, see PRD 016). No retry logic in the client — TanStack Query handles retries at the query level.

**5b. Server State (TanStack Query)**

All bridge API data flows through TanStack Query hooks. Every hook encapsulates:
- The query key (for cache identity and invalidation)
- The fetch function (using the API client)
- Stale time and refetch configuration appropriate to the data type
- Return types matching the API response shapes

| Hook | Endpoint(s) | Stale Time | Refetch |
|------|-------------|------------|---------|
| `useSessions()` | `GET /sessions` | 5s | Every 10s (background) |
| `useSessionDetail(id)` | `GET /sessions/:id/status` | 3s | Every 5s |
| `useSessionEvents(id)` | `GET /sessions/:id/channels/events` | 2s | Every 5s (cursor-based) |
| `useSessionProgress(id)` | `GET /sessions/:id/channels/progress` | 2s | Every 5s (cursor-based) |
| `useAllEvents(since)` | `GET /channels/events` | 2s | Every 5s (cursor-based) |
| `useHealth()` | `GET /health` | 10s | Every 30s |
| `usePoolStats()` | `GET /pool/stats` | 10s | Every 30s |
| `useStrategies()` | `GET /strategies` | 5s | On demand |
| `useTriggers()` | `GET /triggers` | 10s | Every 30s |
| `useTriggerHistory()` | `GET /triggers/history` | 10s | Every 30s |

**Cursor-based polling for channels:** Channel hooks track `last_sequence` from the previous response and pass it as `since_sequence` on the next poll. Only new messages are fetched. The hook appends new messages to the existing cache entry rather than replacing it.

**Mutation hooks:** Write operations (spawn, kill, prompt, trigger enable/disable) use TanStack Query mutations with `onSuccess` callbacks that invalidate related queries:

| Mutation | Endpoint | Invalidates |
|----------|----------|-------------|
| `useSpawnSession()` | `POST /sessions` | `sessions` |
| `useKillSession()` | `DELETE /sessions/:id` | `sessions`, `session-detail` |
| `usePromptSession()` | `POST /sessions/:id/prompt` | `session-detail` |
| `useEnableTrigger()` | `POST /triggers/:id/enable` | `triggers` |
| `useDisableTrigger()` | `POST /triggers/:id/disable` | `triggers` |
| `usePauseTriggers()` | `POST /triggers/pause` | `triggers` |
| `useResumeTriggers()` | `POST /triggers/resume` | `triggers` |
| `useReloadTriggers()` | `POST /triggers/reload` | `triggers` |
| `useExecuteStrategy()` | `POST /strategies/execute` | `strategies` |

**5c. Real-Time Subscriptions (SSE)**

The bridge exposes one SSE endpoint today: `GET /sessions/:id/stream` (raw PTY output for xterm.js). The `useSSE` hook provides a generic SSE subscription pattern:

```typescript
function useSSE<T>(url: string, options: {
  onMessage: (data: T) => void;
  onError?: (error: Event) => void;
  enabled?: boolean;
}): { connected: boolean; error: Event | null };
```

SSE subscriptions are used for:
- **Live terminal output:** `useLiveOutput(sessionId)` connects to `/sessions/:id/stream` and feeds data to xterm.js
- **Query invalidation:** A future SSE endpoint (out of scope for 019.1, noted as extension point) could push event notifications that trigger TanStack Query cache invalidation, replacing polling with push-based updates

Until a push-based SSE event feed exists, the frontend relies on polling intervals (configured per-hook, see table above). This is a known limitation — the existing dashboard already polls on a 5s interval.

**5d. UI State (Zustand)**

Three stores for client-only state. No API data in Zustand — that belongs in TanStack Query.

| Store | State | Persistence |
|-------|-------|-------------|
| `ui-store` | Selected session ID, slide-over open/closed, active slide-over tab, sidebar collapsed, current route params for cross-component coordination | None (ephemeral) |
| `notification-store` | Acknowledged attention item IDs, notification mode (calm/regular/power), snoozed item timestamps | `localStorage` |
| `preference-store` | Theme override (future), timeline density, auto-refresh toggle, default workdir for spawn | `localStorage` |

### Component 6: Routing

```
/app/                              → Dashboard (overview, attention banner, timeline)
/app/sessions                      → Session list (search, filter, sort)
/app/sessions/:id                  → Session detail (metadata, terminal, transcript, channels)
/app/strategies                    → Strategy list + definitions
/app/strategies/:id                → Strategy detail (DAG viz, execution history)
/app/strategies/:id/exec/:execId   → Live execution view (DAG with real-time node status)
/app/triggers                      → Trigger command center (list, fire history, maintenance)
/app/governance                    → Governance overview (council health, RFCs, retros, compliance)
/app/governance/rfcs/:id           → RFC detail + human approval form
/app/analytics                     → Cost, tokens, success rates, subscription meters
/app/registry                      → Method registry browser (tree, method detail, compilation)
/app/commission                    → Commission workflow (create, review, fire, track)
/app/settings                      → Configuration viewer, health, diagnostics
```

**Route-level code splitting:** Every page component is lazy-loaded via `React.lazy()` + `Suspense`. Only the shell (NavBar, PageShell) and the active page are in the initial bundle. A route-level loading skeleton (matching the page layout structure) appears during chunk load.

```typescript
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Sessions = lazy(() => import('./pages/Sessions'));
// ... etc
```

**Route parameters:**
- `:id` for sessions, strategies, and RFCs resolves against the bridge API (session UUID, strategy execution ID, RFC ID)
- `:execId` for strategy executions is the execution-level UUID from `POST /strategies/execute`
- Query parameters for filters: `?status=running`, `?sort=cost`, `?search=impl`

**Navigation model:** The NavBar contains top-level navigation items corresponding to the main route groups. Badge counts on nav items reflect unread/pending items (from `useAttentionItems`). Active item indicated by bioluminescent underline per design language spec.

| Nav Item | Route | Badge Source |
|----------|-------|-------------|
| Dashboard | `/app/` | Attention banner item count |
| Sessions | `/app/sessions` | Running session count |
| Strategies | `/app/strategies` | Executing strategy count |
| Triggers | `/app/triggers` | Active trigger count (yellow if paused) |
| Governance | `/app/governance` | Pending approval count (RFC + essence) |
| Analytics | `/app/analytics` | None |
| Registry | `/app/registry` | None |

Commission and Settings are accessible from the NavBar right-side actions (Commission as a button, Settings as a gear icon), not as primary nav items.

A global CSS rule disables all custom animations when reduced motion is preferred:

```css
@media (prefers-reduced-motion: reduce) {
```

Components that use JavaScript-driven animations (counter tick, sparkline draw) check `window.matchMedia('(prefers-reduced-motion: reduce)').matches` and skip the animation, rendering the final state immediately.

### Component 9: Testing Strategy

| Layer | Tool | Coverage Target | What is Tested |
|-------|------|-----------------|----------------|
| **Unit** | Vitest | 90% on `lib/` | Formatters, API client, type guards, utility functions |
| **Component** | Vitest + React Testing Library | 80% on `components/ui/` and `components/data/` | Rendering, variants, accessibility (ARIA), keyboard interaction, reduced-motion behavior |
| **Integration** | Vitest + MSW (Mock Service Worker) | 70% on hooks | Data fetching hooks return correct shapes, mutations invalidate correct queries, SSE subscription connects and processes events |
| **E2E** | Playwright | Smoke coverage on critical paths | Dashboard loads, session list renders, navigation works, slide-over opens/closes, terminal connects |

**Test file convention:** Tests live next to the source file: `Badge.tsx` + `Badge.test.tsx`. Page-level integration tests live in `pages/__tests__/`.

**MSW for API mocking:** Mock Service Worker intercepts `fetch()` calls during testing, returning fixture data that matches the bridge API response shapes. This avoids coupling tests to a running bridge instance.

**CI integration:** Tests run as part of `npm test` in the bridge workspace. Vitest runs all `*.test.tsx` and `*.test.ts` files. Playwright E2E tests run separately (require a built frontend + running bridge).

### Component 10: Design Language Anti-Pattern Enforcement

The design language and UI guide define explicit anti-patterns. The architecture enforces these structurally:

| Anti-Pattern | Enforcement |
|-------------|-------------|
| Color alone for status | `StatusBadge` component always renders icon + label alongside color. No raw color-only status indicators exist as components. |
| Solar warmth for non-attention items | `--solar` and `--solar-dim` tokens are used only in `AttentionBanner`, HIGH/MEDIUM priority notification components, and `warning` status badge variant. Code review enforces this — no structural guard exists. |
| Show all data at once | Page components follow the three-tier progressive disclosure pattern: page content (Tier 2) is the default, `SlideOverPanel` (Tier 3) opens on explicit click. No page renders Tier 3 detail inline. |
| Animate everything | Animations are defined only on components that represent state change (StatusBadge transitions, ProgressBar fill, TimelineEvent entrance). Layout and text components have no animation. |
| Break chronological order | `Timeline` component sorts events by timestamp. No `sortBy` prop is exposed. Components consuming `Timeline` cannot override the sort. |
| Technical IDs in primary position | Domain components (`SessionCard`, `StrategyCard`, etc.) render `nickname` or `name` as the primary label. UUIDs are available only in Tier 3 detail views. |

## 5. Configuration

| Variable | Default | Description |
|----------|---------|-------------|
| `FRONTEND_ENABLED` | `true` | Master switch — when `false`, `/app/*` routes return 404. Allows running bridge without frontend. |
| `VITE_API_BASE_URL` | `''` (same origin) | Frontend build-time variable. Overrides the API base URL for the fetch client. Empty string means same-origin requests. **In dev mode, all requests (including SSE connections) must go through the Vite proxy — never set this to an absolute origin like `http://localhost:3456`.** The Vite proxy handles forwarding to the bridge. Setting an absolute origin bypasses the proxy and causes CORS issues with SSE. |
| `VITE_SSE_RECONNECT_MS` | `3000` | Delay before SSE reconnect attempt on connection loss. |
| `VITE_POLL_INTERVAL_MS` | `5000` | Default polling interval for TanStack Query hooks. Individual hooks may override. |

No new bridge-side environment variables are introduced. The frontend is served as static assets by the bridge Fastify server using manual file serving with `readFileSync` + MIME type dispatch (matching the existing `GET /viz/*` pattern).

## 6. Out of Scope

- **Page implementations:** Individual pages (Dashboard, Sessions, Strategies, etc.) are delivered by subsequent PRDs (019.2+). This PRD delivers the skeleton, components, hooks, and build pipeline — not the assembled pages.
- **Authentication:** No auth layer. See PRD 016 (Portal as Auth Proxy) for the deferred auth design.
- **WebSocket upgrade:** SSE is sufficient for the current push needs (terminal stream). WebSocket is not needed until bidirectional communication is required (e.g., collaborative editing).
- **Service worker / offline support:** PWA manifest exists but no offline caching. The bridge must be running for the frontend to function.
- **Server-side rendering:** The SPA is client-rendered only. SSR would add complexity with minimal benefit for a local-network developer tool.
- **Storybook component gallery:** Useful but not required for Phase 1. Can be added later as a documentation tool.
- **Internationalization:** English only. No i18n framework.
- **Dark/light theme toggle:** Narrative Flow is dark-only by design. No theme switching.

## 7. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Bundle bloat from xyflow + xterm.js + recharts** — three large visualization libraries | MEDIUM | MEDIUM | Manual chunk splitting isolates each library. xterm.js and xyflow only load on pages that use them (lazy routes). Bundle budget enforced in CI. |
| **Polling overhead at scale** — 10+ hooks polling every 5-10s creates request volume | MEDIUM | LOW | TanStack Query deduplicates concurrent identical requests. Stale time prevents unnecessary refetches. Future SSE event feed (noted as extension point) can replace polling. Hooks disable polling when the browser tab is not visible (`refetchOnWindowFocus`). |
| **Vite proxy vs production path mismatch** — assets work in dev but break in production | LOW | HIGH | Vite `base: '/app/'` ensures all asset paths are relative to the mount point. CI runs a build-and-serve test verifying production paths resolve. |
| **Design token drift** — CSS custom properties diverge from design language spec over time | MEDIUM | LOW | `vidtecci-tokens.css` is the single source. Tailwind config references CSS vars, not hardcoded hex values. Design language changes update one file. |
| **Component API instability** — frequent prop changes during early page development | HIGH | LOW | Tier 1-3 component APIs are stabilized before page development begins (Phase 2 before 019.2). TypeScript interfaces catch breaking changes at compile time. |
| **Existing dashboard regression** — SPA fallback route interferes with existing routes | LOW | HIGH | SPA fallback is scoped to `/app/*` only via a dedicated `app.get('/app/*')` catch-all route (not a global `setNotFoundHandler`). Existing routes are completely independent of this scope. Integration test verifies `/dashboard` still returns server-rendered HTML. |
| **SSE connection limits** — browsers limit concurrent SSE connections per domain (6 in HTTP/1.1) | MEDIUM | MEDIUM | Only one SSE connection is active at a time (the terminal for the currently viewed session). Future multiplexed SSE endpoint could combine multiple streams. HTTP/2 (if enabled) removes the connection limit. |

### Architectural Note

The frontend is a new directory at `packages/bridge/frontend/` with its own `package.json` and build pipeline. It has **zero TypeScript imports** from `@method/bridge`, `@method/core`, or `@method/mcp`. All data flows through HTTP. This preserves DR-03 (core has no transport deps) and ensures the frontend can be developed, tested, and built independently of the server.

The existing `packages/bridge/viz/` app is deprecated by this PRD. Its xyflow components (custom DAG nodes, layout logic) will be migrated into `components/domain/` in a subsequent page PRD (019.X — Strategy views). The viz app continues to be served at `/viz/*` until the migration is complete.
