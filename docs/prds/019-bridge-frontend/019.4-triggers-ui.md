## 1. Purpose and Problem Statement

### Trigger management has no dedicated UI

PRD 018 delivered a complete event trigger system with six trigger types (git_commit, file_watch, schedule, webhook, pty_watcher, channel_event), a debounce engine, concurrency control, and HTTP management endpoints. The existing bridge dashboard shows a basic trigger status table and fire history, but it is a flat table embedded in a multi-panel server-rendered page with no interactivity beyond auto-refresh. There is no way to inspect individual trigger configuration, visualize debounce behavior, review webhook request logs, understand the trigger-to-strategy topology, or perform maintenance operations with confirmation flows.

### Operators need a command center, not a status table

When triggers fire in production, the operator needs to answer questions quickly: Which triggers are active? What fired recently? Why did a trigger fire 4 times instead of 1 — was debouncing working? Did the webhook HMAC validate? Which strategy did this trigger invoke, and how did that execution go? Is the system in maintenance mode? The current UI cannot answer most of these questions without switching to MCP tools or reading raw logs.

### What this PRD delivers

A **Trigger Command Center** page in the Narrative Flow frontend that provides full visibility into the event trigger system. Registered triggers are displayed as interactive cards with type icons, status indicators, and fire rate sparklines. Clicking a trigger opens a slide-over detail panel with configuration, fire history, debounce visualization, and an enable/disable toggle. A fire history timeline shows chronological trigger fires across all triggers. Maintenance mode is surfaced as a prominent banner with one-click resume. A topology view shows the full chain from triggers through strategies to execution nodes. Webhook triggers get a dedicated request log showing payload previews and HMAC validation results.

## 3. Components

### Component 1: Trigger Card List

A vertical list of cards, one per registered trigger. This is the Tier 2 (scan) view — the operator identifies what exists and what needs attention within 5 seconds.

**Card contents:**

| Element | Source | Typography | Position |
|---------|--------|-----------|----------|
| Type icon | Trigger type from `GET /triggers` | 36px rounded square, type-colored background | Left |
| Trigger ID | `trigger_id` field | JetBrains Mono, `--text-sm`, `--text` | Header, after icon |
| Type arrow strategy | `type` + `strategy_id` | `--text-xs`, `--text-dim`; strategy name in `--bio` | Below trigger ID |
| Status dot | `enabled` + global `paused` | 8px circle, animated pulse if active | Header, right |
| Status label | Derived | `--text-xs`, color-coded | Right of status dot |
| Config summary | Type-specific fields (2x2 grid) | Label: `--text-xs`, `--text-muted`, uppercase; Value: JetBrains Mono, `--text-sm`, `--text-dim` | Card body |
| Fire count | `stats.total_fires` | JetBrains Mono, `--text-xs` | Footer, left |
| Error count | `stats.errors` | JetBrains Mono, `--text-xs`, `--error` if > 0 | Footer |
| Fire rate sparkline | Computed from fire history | 24 bars, 3px wide, 20px height, `--bio` fill | Footer, right-aligned |

**Type-specific config summary fields:**

| Type | Field 1 | Field 2 | Field 3 | Field 4 |
|------|---------|---------|---------|---------|
| `git_commit` | Branch pattern | Debounce | Max concurrent | Last fired |
| `file_watch` | Watch paths | Debounce | Max batch size | Last fired |
| `schedule` | Cron expression | Human-readable interval | Next fire (computed) | Last fired |
| `webhook` | Webhook path | HMAC configured | Filter expression | Last fired |
| `pty_watcher` | Pattern category | Condition expression | Debounce | Last fired |
| `channel_event` | Event types | Filter expression | Debounce | Last fired |

**Type icon color mapping:**

| Type | Icon | Background | Color |
|------|------|-----------|-------|
| `git_commit` | Branch glyph | `--nebular-dim` | `--nebular` |
| `file_watch` | Folder glyph | `--solar-dim` | `--solar` |
| `schedule` | Clock glyph | `--bio-dim` | `--bio` |
| `webhook` | Arrow-in glyph | `rgba(0, 229, 204, 0.15)` | `--cyan` |
| `pty_watcher` | Terminal glyph | `--error-dim` | `--error` |
| `channel_event` | Message glyph | `--nebular-dim` | `--nebular` |

**Card states:**

| State | Visual Treatment |
|-------|-----------------|
| Active, healthy | Status dot: `--bio` with `pulse-glow-sm` animation. Border: default |
| Active, recent error | Status dot: `--bio`. Error count: `--error`. Border: default |

- Only one card selected at a time
- Click outside / Escape / slide-over close: deselects card, URL returns to `/triggers`
- Cards enter with staggered `slide-in-left` animation (100ms per card)
- Hover: `translateY(-2px)` lift, border brightens to `--border-hover`

**Data source:** `GET /triggers` endpoint. Poll every 10 seconds or use SSE when available.

### Component 2: Trigger Detail Slide-Over

The Tier 3 (deep dive) view. Opens from the right when a trigger card is clicked. Contains tabbed content for different aspects of the trigger.

**Header:**
- Type label: "TRIGGER DETAIL" in `--text-xs`, JetBrains Mono, `--text-muted`, uppercase
- Trigger ID: Space Grotesk, `--text-lg`, `--text`
- Close button: X glyph, `--text-dim`, hover brightens

**Tabs:**

| Tab | Content | Visible for |
|-----|---------|-------------|
| Config | Full configuration + enable/disable toggle | All triggers |
| Fire History | Detailed fire history table | All triggers |
| Debounce | Debounce visualization chart | All triggers with debounce |
| Webhook | Request log with payload + HMAC status | `webhook` type only |

#### Tab: Config

**Enable/disable toggle:** A toggle switch row at the top of the config tab. Toggling calls `POST /triggers/:id/enable` or `POST /triggers/:id/disable`. The toggle reflects the current `enabled` state. When the trigger is disabled, the toggle slides left and the track color changes from `--bio` to `--border-hover`.

**Configuration fields:** Rendered as a structured view (label-value pairs). All fields from the trigger's YAML config are shown.

| Field | Type | Notes |
|-------|------|-------|
| Type | All | Trigger type name |
| Strategy | All | Strategy ID, rendered as a clickable link (`--bio`) navigating to `/strategies/:id` |
| Enabled | All | Current enabled state |
| Status | All | Derived: active / disabled / paused / error |
| Type-specific fields | Varies | Branch pattern, paths, cron, webhook path, pattern, event_types, filter, etc. |
| Debounce window | All with debounce | Duration in human-readable form (e.g., "15s leading") |
| Max concurrent | All | Concurrency limit |
| Max batch size | All with debounce | Hard cap before forced fire |

**Actions footer:**
- "View Strategy" button (`btn-secondary`): navigates to `/strategies/:strategy_id`

#### Tab: Fire History

A table of trigger fire events for this specific trigger, ordered most recent first.

**Table columns:**

| Column | Width | Content | Typography |
|--------|-------|---------|-----------|
| Time | 80px | Timestamp (HH:MM, or "Yest HH:MM" for yesterday, or date for older) | JetBrains Mono, `--text-xs`, `--text-dim` |
| Debounce | 50px | Raw events -> fires (e.g., "3->1") | JetBrains Mono, `--text-xs`, `--solar` |
| Execution | flex | Execution ID (truncated, clickable link to strategy execution) | JetBrains Mono, `--text-xs`, `--text-muted` |
| Cost | 60px | Execution cost (e.g., "$0.34") | JetBrains Mono, `--text-xs`, `--text-dim`, right-aligned |

**Row styling:** Each row is a compact bar with `--void` background, `--border` border, 6px border-radius. Rows stack vertically with `--space-2` gap.

**Data source:** `GET /triggers/history?trigger_id={id}` filtered by trigger ID. The `trigger_id` query parameter must be supported by the history endpoint to filter results to a single trigger (backend prerequisite if not already supported). Show last 20 fires. "Load more" link at the bottom to paginate.

**Execution link:** Clicking an execution ID navigates to `/strategies/:strategyId/exec/:executionId` (Strategy UI, PRD 019.3).

#### Tab: Debounce

A bar chart visualization showing raw events versus debounced fires per time window, answering the question: "Is debouncing doing its job?"

**Chart structure:**
- X axis: time windows (last 12 hours, divided into equal buckets)
- Y axis: event count (auto-scaled)
- Each time bucket has two stacked bars:
  - **Raw events** (top): `rgba(138, 155, 176, 0.2)` — how many raw events the watcher received
  - **Debounced fires** (bottom): `--bio` at 70% opacity — how many actual trigger fires occurred
- Bar groups are 100% width of the chart, with 3px gap between groups
- Chart height: 60px
- Chart padding: `--space-3` vertical

**Legend:** Below the chart, a horizontal legend with two items:
- Gray swatch + "Raw events"
- Bio swatch + "Debounced fires"

Font: `--text-xs`, `--text-muted`.

**Debounce summary stats** (above chart):
- Total raw events (last 12h)
- Total debounced fires (last 12h)
- Collapse ratio (e.g., "4.2:1" meaning 4.2 raw events per fire)
- Current debounce window (from config)

**Data source:** Computed from `GET /triggers/history` by bucketing fire events into time windows. Raw events are represented by the `debounced_count` field on each fire event — this is the number of raw events that were collapsed into that single fire. The chart shows collapse ratios per fire, not independent raw event timestamps (raw events are not individually timestamped in the history). For each time bucket: "raw events" = sum of `debounced_count` for all fires in the bucket; "debounced fires" = count of fires in the bucket.

**Animation:** Bars draw upward with `ease-out` animation (1000ms) on tab activation.

#### Tab: Webhook (webhook type only)

A request log for webhook triggers, showing recent incoming HTTP requests with their validation status.

**Table columns:**

| Column | Width | Content | Typography |
|--------|-------|---------|-----------|
| Time | 80px | Request timestamp | JetBrains Mono, `--text-xs`, `--text-dim` |
| Method | 50px | HTTP method (POST, PUT, etc.) | JetBrains Mono, `--text-xs`, badge-styled |
| HMAC | 60px | Validation status (pass/fail/none) | Badge: pass = `--bio`/`--bio-dim`, fail = `--error`/`--error-dim`, none = `--text-muted` |
| Filter | 60px | Filter expression result (pass/reject/N/A) | Badge: pass = `--bio`, reject = `--solar`, N/A = `--text-muted` |
| Payload | flex | Truncated payload preview (first 80 chars of JSON) | JetBrains Mono, `--text-xs`, `--text-muted` |

**Row expansion:** Clicking a webhook request row expands it to show the full payload (JSON-formatted, syntax highlighted) and full request headers.

**Data source:** This requires a new data endpoint. Webhook triggers do not currently store request history. See Section 6 (API Extensions) for the proposed `GET /triggers/:id/webhook-log` endpoint.

### Component 3: Fire History Timeline

A chronological feed of trigger fires across all triggers, displayed below the trigger card list on the main Triggers page. This is the narrative thread — the operator reads what happened in time order.

**Structure:** Standard Narrative Flow timeline (vertical line on the left, colored dots per event type, time labels, event cards).

**Timeline dot colors:** Follow the trigger type icon color mapping:
- `git_commit`: `--solar` (matches the "Trigger fired" event type color from the design language)
- `file_watch`: `--solar`
- `schedule`: `--bio`
- `webhook`: `--cyan`
- `pty_watcher`: `--error`
- `channel_event`: `--nebular`

**Event card contents:**
- Time: HH:MM in JetBrains Mono, `--text-dim`
- Type badge: colored pill with trigger type icon and name (matches card type badge)
- Arrow + Strategy name: `--text-muted` arrow, strategy ID in `--bio`, JetBrains Mono
- Detail line: human-readable summary (e.g., "debounced 3 commits -> 1 fire", "1 file created", "cron")

**Time grouping:** Events are grouped by time period with sticky section headers: "Today", "Yesterday", dates for older events. Headers use `--text-xs`, `--text-muted`, uppercase, Space Grotesk.

**Interactions:**
- Click event card: opens the trigger's slide-over detail panel on the Fire History tab
- Cards enter with staggered animation (100ms per item, starting after the trigger card animations complete)
- Hover: subtle card lift + border glow

**Data source:** `GET /triggers/history` with default limit of 50 events. "Load more" at the bottom of each time group.

### Component 4: Maintenance Mode Banner

A prominent attention banner displayed at the top of the Triggers page content area when triggers are globally paused (`paused: true` from `GET /triggers`).

**Design:** Follows the Narrative Flow Attention Banner pattern:
- Full-width card below nav
- Left border: 3px solid `--solar`
- Background: `--solar-dim`
- Icon: warning triangle, `--solar`
- Text: "Triggers Paused -- Maintenance Mode" in `--text-sm`, `--solar`, font-weight 500
- Resume button: solid `--solar` background, `--void` text, `--text-xs`, font-weight 600

**Confirmation flow:** Clicking "Resume" shows a confirmation tooltip or inline prompt: "Resume all N active triggers?" with Confirm/Cancel buttons. On confirm, calls `POST /triggers/resume`. On success, the banner disappears with a slide-up animation.

**Pause action:** The toolbar (Section 3.5) contains the Pause button. Clicking it shows a confirmation: "Pause all N active triggers? Trigger fires will be suppressed until resumed." On confirm, calls `POST /triggers/pause`. On success, the maintenance banner appears with slide-down animation.

**Animation:** The banner uses `slide-down` entrance (500ms ease-out) and `slide-up` exit (300ms ease-in), following the design language spec.

### Component 5: Trigger Topology View

A visual representation of the trigger-to-strategy-to-node chain, answering the question: "What is connected to what?" (UC-78).

**Access:** A "Topology" button in the triggers toolbar. Clicking it opens a full-page overlay or replaces the trigger list with the topology view (toggle behavior).

**Structure:** A tree or layered graph with three columns:

```
Triggers              Strategies           Nodes
+-----------+         +-----------+       +-----------+
| git_commit|-------->|S-CODE-    |------>| section   |
|           |         |  REVIEW   |------>| implement |
+-----------+         +-----------+------>| review    |
                                  |------>| gate      |
+-----------+         +-----------+
| file_watch|-------->|S-PERF-    |------>| detect    |
|           |         |  FILE-    |------>| analyze   |
+-----------+         |  WATCH    |
                      +-----------+
+-----------+
| schedule  |-------->|S-HEALTH-  |------>| check     |
|           |         |  CHECK    |------>| report    |
+-----------+         +-----------+
```

**Node rendering:**
- Trigger nodes: colored by trigger type (same icon color mapping), show trigger ID
- Strategy nodes: `--bio` accent, show strategy ID
- DAG nodes: `--text-dim`, show node ID and type (methodology/script/gate)
- Edges: lines connecting triggers to strategies, and strategies to their top-level DAG nodes

**Interactions:**
- Click trigger node: navigates to trigger detail (opens slide-over)
- Click strategy node: navigates to `/strategies/:id`
- Click DAG node: navigates to `/strategies/:id` with node highlighted
- Hover: highlight connected path (trigger -> strategy -> nodes)

**Data source:** Assembled client-side from `GET /triggers` (trigger -> strategy mapping) and `GET /api/strategies/definitions` (from PRD 019.3 — provides strategy node lists including `depends_on` edges). The definitions endpoint returns all strategies with their node arrays, which is sufficient to build the topology graph without needing per-execution DAG endpoints. No execution data is required for topology rendering.

**Implementation note:** The topology view can be rendered with a simple CSS grid or flexbox layout for Phase 1. If the number of triggers/strategies grows, consider using xyflow (already a project dependency for the Strategy visualizer) for a proper graph layout.

## 4. Data Flow

### Polling Strategy

The Triggers page polls two endpoints:

| Endpoint | Interval | Purpose |
|----------|----------|---------|
| `GET /triggers` | 10s | Trigger list, status, stats, paused flag |
| `GET /triggers/history?limit=50` | 10s | Fire history for timeline |

When SSE is available (future PRD 019.1 infrastructure), replace polling with real-time event push. Trigger fires are emitted as `trigger_fired` events on the bridge's channel system (PRD 018 Component 8), which can feed SSE directly.

### State Management

Trigger UI state is managed in a dedicated store (zustand slice or similar):

```typescript
interface TriggerUIState {
  triggers: TriggerListItem[];
  history: TriggerFireEvent[];
  paused: boolean;
  selectedTriggerId: string | null;
  topologyVisible: boolean;
  webhookLogs: Record<string, WebhookLogEntry[]>;
  loading: boolean;
  lastUpdated: string | null;
}
```

### Optimistic Updates

Enable/disable toggle and pause/resume actions apply optimistic UI updates immediately, then confirm with the server response. On server error, the UI reverts and shows an error toast.

## 5. Use Case Mapping

### UC-10: View Trigger Status and Fire History

**Flow:**
1. Human navigates to `/triggers` (or clicks "Triggers" in nav)
2. Trigger card list loads with current status, stats, and sparklines
3. Fire history timeline loads below the cards
4. Human scans cards to identify active/disabled/errored triggers
5. Human reads timeline to see recent fire activity

**Components:** Trigger Card List (C1), Fire History Timeline (C3)

### UC-78: View Trigger Topology

**Flow:**
1. Human clicks "Topology" button in the toolbar
2. The topology view replaces the trigger list, showing trigger -> strategy -> node chains
3. Human reads left-to-right: which triggers invoke which strategies, which strategies contain which nodes
4. Human clicks a strategy node to navigate to the Strategy detail page
5. Human clicks "Topology" again (or presses Escape) to return to the trigger list

**Components:** Trigger Topology View (C5), Toolbar (C6)

### UC-79: View Debounce Visualization

**Flow:**
1. Human clicks a trigger card to open the slide-over
2. Human clicks the "Debounce" tab
3. The bar chart renders showing raw events vs debounced fires per time bucket
4. Human reads the collapse ratio to assess debounce effectiveness
5. If the ratio is near 1:1, debouncing is too aggressive (not collapsing enough). If it is very high, debouncing is effective but the human may want to check if events are being lost

**Components:** Trigger Detail Slide-Over (C2, Debounce tab)

### UC-80: View Webhook Request Log

**Flow:**
1. Human clicks a webhook trigger card to open the slide-over
2. Human clicks the "Webhook" tab (visible only for webhook type)
3. The request log shows recent incoming webhooks with method, HMAC status, filter result, and payload preview
4. Human clicks a row to expand full payload and headers
5. Human checks HMAC column: all "pass" = healthy; any "fail" = investigate sender or secret mismatch

**Components:** Trigger Detail Slide-Over (C2, Webhook tab)

### UC-81: View Strategy Execution Comparison

**Flow:**
1. Human clicks a trigger's fire history row with an execution ID
2. Navigation goes to `/strategies/:id/exec/:eid` (Strategy UI, PRD 019.3)
3. The human uses the Strategy UI's execution comparison feature to compare this trigger-invoked execution with a previous one

**Components:** Trigger Detail Slide-Over (C2, Fire History tab) -- handoff to Strategy UI

**Note:** UC-81 is primarily fulfilled by the Strategy UI (PRD 019.3). The Triggers UI's contribution is providing the navigation path from trigger fire -> execution -> comparison.

## 6. API Extensions

### Existing Endpoints (PRD 018)

| Endpoint | Used by | Changes Required |
|----------|---------|-----------------|
| `GET /triggers` | Trigger Card List, Maintenance Banner | **Must be extended** to return `trigger_config` fields (branch_pattern, paths, cron, debounce_ms, max_concurrent, max_batch_size, webhook_path, hmac_configured, filter_expression, etc.) for each trigger. Currently returns status and stats but not the full configuration needed for the trigger card config summary and the slide-over Config tab. This is a backend prerequisite. |
| `GET /triggers/history` | Fire History Timeline, Debounce tab computation | No changes needed |
| `POST /triggers/:id/enable` | Config tab toggle | No changes needed |
| `POST /triggers/:id/disable` | Config tab toggle | No changes needed |
| `POST /triggers/pause` | Toolbar pause button | No changes needed |
| `POST /triggers/resume` | Toolbar resume button, Maintenance Banner | No changes needed |
| `POST /triggers/reload` | Toolbar reload button | No changes needed |

### New Backend Prerequisites

#### `GET /triggers/:id`

Returns the full detail for a single trigger, including all configuration fields, stats, and recent fire history. This endpoint does not exist in the current PRD 018 implementation and must be added as a backend prerequisite.

**Response shape:** Same as a single item from `GET /triggers`, but with the full `trigger_config` object and optionally a `recent_fires` array (last 10 fires).

**Used by:** Trigger Detail Slide-Over (Config tab), deep link routing (`/triggers/:id`).

### New Endpoint Required

#### `GET /triggers/:id/webhook-log`

Returns recent webhook requests for a specific webhook trigger.

**Response:**
```json
{
  "trigger_id": "S-CODE-REVIEW:webhook:1",
  "requests": [
    {
      "timestamp": "2026-03-20T14:28:00Z",
      "method": "POST",
      "hmac_status": "pass",
      "filter_result": "pass",
      "payload_preview": "{\"action\":\"completed\",\"repo\":\"method\",...}",
      "headers": { "content-type": "application/json", "x-hub-signature-256": "sha256=..." },
      "payload_size_bytes": 1420
    }
  ],
  "count": 5
}
```

**Implementation:** The `WebhookTrigger` class (`packages/bridge/src/triggers/webhook-trigger.ts`) needs a ring buffer of recent requests (last 50). Each call to `handleWebhook()` appends to the buffer. The route handler in `trigger-routes.ts` exposes this via `GET /triggers/:id/webhook-log`.

**Query parameters:**
- `limit` (optional, default 20, max 50): number of recent requests to return

### Sparkline Computation

Fire rate sparklines are computed client-side from the fire history data. The algorithm:

1. Take the last 24 hours of fire history for this trigger (filter `GET /triggers/history` by trigger ID and time)
2. Divide into 24 one-hour buckets
3. Count fires per bucket
4. Normalize to the sparkline bar height range (2px minimum, 20px maximum)
5. Buckets with zero fires render as `sparkline-bar zero` (2px, dimmed)

## 7. Responsive Behavior

| Breakpoint | Layout Change |
|-----------|--------------|
| > 1200px | Full layout: trigger cards + slide-over side by side |
| 768-1200px | Trigger cards full-width; slide-over overlays content (backdrop blur) |
| < 768px | Trigger cards stack full-width; slide-over becomes full-screen push; toolbar wraps to 2 rows; topology view uses vertical layout |

### Mobile Considerations

- Trigger cards become full-width single-column
- Sparklines remain visible (they are compact enough for mobile)
- The config summary grid switches from 2-column to 1-column
- Slide-over becomes a full-screen page push (back arrow replaces close X)
- Fire history timeline remains single-column (natural mobile layout)
- Topology view uses a vertical tree layout instead of horizontal columns

## 9. Accessibility

| Concern | Approach |
|---------|----------|
| Color alone for status | Every status has icon + text label + color. Status dot is paired with a label ("Active", "Paused", "Disabled") |
| Keyboard navigation | Tab through trigger cards, Enter to select, Escape to deselect/close slide-over. Arrow keys within slide-over tabs |
| Screen reader | Trigger cards have `role="button"` and `aria-label` with trigger ID and status. Slide-over uses `role="dialog"` with `aria-labelledby` |
| HMAC status badges | "Pass" / "Fail" / "None" text labels, not just color |
| Sparklines | `aria-label` with human-readable summary (e.g., "3 fires in last 24 hours, peak at 14:00") |
| Reduced motion | All animations disabled. Status dot pulse replaced with static indicator |
| Focus trap | Slide-over traps focus when open. Escape closes it. Focus returns to the triggering card |

## 10. Implementation Phases

### Phase 1: Core Trigger List + Detail (Minimal Viable)

**Deliverables:**
- Trigger Card List component with type icons, status, stats, sparklines
- Trigger Detail Slide-Over with Config tab and enable/disable toggle
- Fire History Timeline component
- Maintenance Mode Banner with pause/resume
- Toolbar with Reload and Pause/Resume buttons
- Route: `/triggers` and `/triggers/:id`
- Polling: `GET /triggers` and `GET /triggers/history` at 10s intervals

**Why first:** This covers UC-10 (the primary use case) completely. The operator can see what triggers exist, their status, recent fires, and perform basic management (enable/disable, pause/resume, reload).

### Phase 2: Debounce + Webhook Tabs

**Backend prerequisite (must be completed before Phase 2 frontend work):** The `WebhookTrigger` class needs a ring buffer of recent requests (last 50) added to its `handleWebhook()` method, and the `GET /triggers/:id/webhook-log` route must be registered in `trigger-routes.ts`. Without this backend delta, the Webhook tab has no data source.

**Deliverables:**
- Debounce visualization tab (bar chart, collapse ratio, summary stats)
- Webhook request log tab (requires `GET /triggers/:id/webhook-log` endpoint — backend prerequisite above)
- `WebhookTrigger` ring buffer implementation in the bridge (backend delta)
- Expanded fire history with execution links

**Why second:** UC-79 and UC-80 are diagnostic use cases — needed when something goes wrong, not for routine monitoring. The debounce tab requires client-side computation. The webhook tab requires a new bridge endpoint.

### Phase 3: Topology View

**Deliverables:**
- Trigger Topology View component (three-column layout)
- Toolbar "Topology" toggle button
- Strategy DAG data fetching for topology edges
- Interactive navigation from topology nodes

**Why third:** UC-78 is an orientation use case — the operator uses it when first setting up triggers or when the trigger landscape changes. It requires fetching data from multiple endpoints and assembling a cross-cutting view. Lower priority than direct monitoring.

## 11. Success Criteria

1. **Trigger list renders correctly.** All registered triggers from `GET /triggers` appear as cards with correct type icons, status indicators, stats, and sparklines. Cards update on 10s polling cycle.

2. **Detail slide-over shows full configuration.** Clicking any trigger card opens the slide-over with the Config tab showing all YAML-defined configuration fields. The enable/disable toggle calls the correct endpoint and reflects the new state.

3. **Fire history timeline is chronological.** Trigger fires from `GET /triggers/history` render in time order with correct type badges, strategy names, and debounce summaries. Time grouping headers ("Today", "Yesterday") are accurate.

4. **Maintenance mode is prominent.** When `paused: true`, the maintenance banner is visible at the top of the page. Clicking Resume triggers a confirmation flow, then calls `POST /triggers/resume` and hides the banner.

5. **Debounce visualization is informative.** The bar chart shows a clear difference between raw events and debounced fires. The collapse ratio is computed and displayed. The operator can determine at a glance whether debouncing is effective.

6. **Webhook log shows validation status.** For webhook triggers, the Webhook tab shows recent requests with HMAC pass/fail badges and filter results. Clicking a row expands the full payload.

7. **Topology shows the full chain.** The topology view correctly maps all triggers to their strategies and strategies to their DAG nodes. Clicking nodes navigates to the correct detail pages.

8. **Reload shows results.** The Reload button calls `POST /triggers/reload` and displays a toast with the added/updated/removed/errors summary.

9. **Responsive on mobile.** At < 768px, trigger cards are full-width, slide-over becomes full-screen, toolbar wraps, and all content remains readable and interactive.

10. **No regression on existing pages.** Adding the Triggers page does not break navigation, state management, or polling on other pages (Dashboard, Sessions, Strategies).

## 12. Relationship to Other PRDs

| PRD | Relationship |
|-----|-------------|
| **PRD 018** (Event Triggers) | The Triggers UI is the frontend for PRD 018's HTTP API. All trigger management endpoints (`GET /triggers`, `POST /triggers/:id/enable`, etc.) are consumed directly. One new endpoint (`GET /triggers/:id/webhook-log`) is proposed. |
| **PRD 017** (Strategy Pipelines) | Trigger fires invoke strategy executions. The Triggers UI links to strategy detail pages and execution views. The topology view fetches strategy DAG data. |
| **PRD 019.1** (Foundation) | The Triggers page depends on the shared navigation, routing, state management, and design system components delivered by PRD 019.1. |
| **PRD 019.3** (Strategies UI) | UC-81 (execution comparison) is fulfilled by the Strategy UI. The Triggers UI provides navigation from fire history -> execution detail. Strategy links in trigger cards navigate to the Strategy detail page. |
| **PRD 008** (Agent Visibility) | Trigger fires emit `trigger_fired` events to the bridge channel system. When SSE replaces polling, these events will drive real-time updates on the Triggers page. |
| **PRD 010** (PTY Auto-Detection) | The `pty_watcher` trigger type hooks into PTY observation callbacks. The Triggers UI renders pty_watcher triggers with their pattern and condition configuration. |
