# PRD 014 — Sub-Agent Scope Enforcement

**Status:** Draft
**Date:** 2026-03-15
**Scope:** Metadata-driven path constraints for spawned agents, pre-commit hook generation, PTY watcher scope violation detection, commission skill integration
**Depends on:** PRD 006 (recursive orchestration + worktree isolation), PRD 010 (PTY watcher infrastructure)
**Evidence:** OBS-05 (sub-agents commit out-of-scope files), project-card `role_notes.impl_sub_agent` (prompt-level mitigation insufficient), RFC #3 recommendations
**Council:** Pending

## 1. Purpose and Problem Statement

### Sub-agents routinely modify files outside their declared scope

OBS-05 documents this across multiple commissions: agents spawned for a specific task modify files they were never meant to touch. Examples range from harmless (an architecture docs agent committing council log updates) to destructive (a sub-agent deleting M1-COUNCIL registry content during a PRD 004 implementation).

Current mitigations are all prompt-level:

1. **Project card `role_notes.impl_sub_agent`** — warns agents to stay in scope
2. **CLAUDE.md sub-agent guidelines** — tells agents not to commit outside task scope
3. **Commission skill Section 9** — requires explicit file-scope listing per sub-agent

Prompt-level mitigations are demonstrably insufficient. OBS-13 proved this pattern for channel reporting (agents ignore prompt instructions when they don't affect task completion). The same holds for scope constraints — agents optimize for completing the task, and if touching an out-of-scope file helps them get there, they do it. The agent is not being malicious; it simply doesn't have an enforcement mechanism beyond its own judgment.

### The solution: infrastructure-level enforcement

The bridge already controls the environment agents execute in. Worktree isolation (PRD 006) gives each agent its own git worktree. PTY watcher (PRD 010) observes every file operation in real time. These two capabilities, combined with a new `allowed_paths` metadata field, enable three layers of scope enforcement:

1. **Pre-commit hook** — git-level hard block: commits that include files outside `allowed_paths` are rejected
2. **PTY watcher detection** — real-time soft signal: `Write`/`Edit` operations on out-of-scope files emit `scope_violation` events to channels
3. **Commission integration** — the commission skill populates `allowed_paths` automatically from the task's file-scope specification

No agent changes. No prompt modifications. The bridge becomes the enforcer.

## 2. Components

### Component 1: `allowed_paths` Metadata

A new optional field in session metadata, accepted by `bridge_spawn` and propagated through the session lifecycle.

```typescript
interface ScopeConstraint {
  /** Glob patterns of files the agent is allowed to modify. */
  allowed_paths: string[];
  /**
   * Enforcement mode:
   * - 'enforce': pre-commit hook rejects out-of-scope commits (default)
   * - 'warn': PTY watcher emits events but commits are not blocked
   */
  mode: 'enforce' | 'warn';
}
```

**Glob semantics:** Patterns follow `.gitignore`-style matching via `git diff --name-only` filtering. Examples:

| Pattern | Matches |
|---------|---------|
| `packages/bridge/src/**` | All files under `packages/bridge/src/` |
| `packages/bridge/src/__tests__/**` | Test files only |
| `docs/prds/*.md` | PRD markdown files |
| `*.ts` | Any TypeScript file at any depth |
| `packages/core/**` `packages/mcp/**` | Multiple package scopes |

An empty `allowed_paths` array means "no scope constraint" (backwards compatible — all existing sessions continue to work without restriction).

### Component 2: Pre-Commit Hook Generator

When a session is spawned with `allowed_paths` and `mode: 'enforce'` (or mode is omitted, since enforce is default), the bridge generates a git pre-commit hook in the session's worktree.

**Hook placement:** `<worktree_path>/.git/hooks/pre-commit` (or, for linked worktrees, the worktree-specific hooks directory).

**Hook logic:**

```bash
#!/usr/bin/env bash
# Auto-generated by pv-method bridge (PRD 014)
# Scope enforcement for session: <session_id>
# Allowed paths:
#   <pattern_1>
#   <pattern_2>
#   ...

ALLOWED_PATTERNS=(<pattern_1> <pattern_2> ...)

# Get staged files
STAGED=$(git diff --cached --name-only)

for file in $STAGED; do
  allowed=false
  for pattern in "${ALLOWED_PATTERNS[@]}"; do
    # Use bash glob matching (extglob)
    if [[ "$file" == $pattern ]]; then
      allowed=true
      break
    fi
  done
  if [ "$allowed" = false ]; then
    echo "SCOPE VIOLATION: '$file' is not in allowed_paths for this session."
    echo "Allowed patterns: ${ALLOWED_PATTERNS[*]}"
    echo "Remove the file from staging (git reset HEAD '$file') or request scope extension."
    exit 1
  fi
done
```

**Key design decisions:**

- **Bash, not Node:** The hook must be self-contained with no runtime dependencies. Git hooks execute in a bare shell environment.
- **Worktree-only:** Pre-commit hooks are only generated when `isolation: 'worktree'`. In shared mode, modifying `.git/hooks/` would affect all sessions. If `allowed_paths` is provided without worktree isolation, the bridge falls back to `mode: 'warn'` and logs a warning.
- **Git glob matching:** Patterns use the same semantics as `.gitignore` — `**` matches any depth, `*` matches within a directory. The hook implementation uses bash `extglob` to approximate this. For complex patterns, `git diff --cached --name-only | grep -E` is the fallback.
- **Non-destructive on failure:** If the hook rejects a commit, Claude Code receives the rejection message via stderr. The agent can then `git reset HEAD <file>` and retry. The PTY watcher's error pattern (Pattern 7) will also detect the hook rejection.

**Behavior:**

| Agent Action | In-Scope Path | Out-of-Scope Path |
|-------------|---------------|-------------------|
| `Read` | Normal `file_activity` | Normal `file_activity` (reads are never violations) |
| `Write` | Normal `file_activity` | `scope_violation` event emitted |
| `Edit` | Normal `file_activity` | `scope_violation` event emitted |
| `git commit` | Normal `git_commit` | Pre-commit hook blocks (enforce) or `scope_violation` event (warn) |

**Channel emission:**

| Channel | Type | Content |
|---------|------|---------|
| `events` | `scope_violation` | `{ path: string, operation: 'write' \| 'edit', allowed_patterns: string[] }` |

**Push notification:** `scope_violation` events trigger parent push notifications (same as `error` and `escalation` events in PRD 008). This allows the orchestrator to intervene immediately when a sub-agent goes out of scope.

### Component 4: `bridge_spawn` API Extension

The `bridge_spawn` tool gains two new optional parameters:

```typescript
// In MCP tool schema
{
  allowed_paths: {
    type: "array",
    items: { type: "string" },
    description: "Glob patterns of files this agent is allowed to modify. Empty = no constraint. Requires isolation: 'worktree' for enforcement mode.",
  },
  scope_mode: {
    type: "string",
    enum: ["enforce", "warn"],
    description: "Scope enforcement mode. 'enforce' installs a pre-commit hook (requires worktree). 'warn' emits events only. Default: 'enforce'.",
  },
}
```

**Spawn flow with scope enforcement:**

```
bridge_spawn(workdir, isolation: 'worktree', allowed_paths: ['packages/bridge/**'])
  │
  ├─ 1. Create worktree (existing PRD 006 logic)
  ├─ 2. Store allowed_paths in session metadata
  ├─ 3. Generate pre-commit hook in worktree (Component 2)
  ├─ 4. Configure PTY watcher with scope context (Component 3)
  └─ 5. Spawn agent in worktree (existing logic)
```

### Component 5: Commission Skill Integration

The `/commission` skill's task specification already includes file-scope listings (Section 9). PRD 014 extends this by:

1. **Auto-populating `allowed_paths`** from the commission's file-scope specification when generating the orchestrator prompt
2. **Embedding scope metadata** in the `bridge_spawn` call template that the commission prompt includes

The commission skill produces orchestrator prompts that spawn sub-agents. When the skill knows the file scope, it writes the spawn instruction as:

```
bridge_spawn(
  workdir: "/path/to/project",
  isolation: "worktree",
  allowed_paths: ["packages/bridge/src/**", "packages/bridge/src/__tests__/**"],
  scope_mode: "enforce",
  initial_prompt: "..."
)
```

This is a prompt-level integration — the commission skill template includes `allowed_paths` in the spawn call it generates. The bridge then enforces it mechanically.

## 3. Enforcement Matrix

Three enforcement layers work independently. Any layer can catch a scope violation even if others fail.

| Layer | Mechanism | Timing | Hard/Soft | Works Without Worktree |
|-------|-----------|--------|-----------|----------------------|
| **Pre-commit hook** | Git hook in worktree | At `git commit` | Hard block | No — requires worktree |
| **PTY watcher** | Pattern 8 on Write/Edit | Real-time, on file operation | Soft signal (event) | Yes — metadata-based |
| **Commission prompt** | `allowed_paths` in spawn call | At spawn time | Template-level | N/A — generates the call |

**Why three layers:**

- The pre-commit hook is the only hard block — it physically prevents out-of-scope commits. But it only fires at commit time, not when the agent writes the file. An agent can write 10 out-of-scope files, and the hook only fires when it tries to commit them. By then the agent has wasted time.
- The PTY watcher catches violations as they happen — the moment the agent writes to an out-of-scope file, the orchestrator gets a push notification. This enables early intervention ("stop, you're going out of scope") before the agent digs deeper.
- The commission integration ensures `allowed_paths` is populated in the first place. Without it, orchestrators would need to manually specify paths — and they won't, because prompt-level diligence is exactly what OBS-05 proves doesn't work.

## 4. Implementation Order

### Phase 1: Metadata + Pre-Commit Hook

**Deliverables:**
- `allowed_paths` and `scope_mode` accepted by pool `create()` and stored in session metadata
- Pre-commit hook generation function in `packages/bridge/src/scope-hook.ts`
- Hook installed after worktree creation in pool's create flow
- `bridge_spawn` MCP tool schema updated with new parameters
- Unit tests: hook generation with various patterns, hook script correctness
- Integration test: spawn with `allowed_paths`, attempt out-of-scope commit, verify rejection

**Why first:** The pre-commit hook is the hard enforcement layer. Without it, everything else is advisory. The hook is also the simplest component — it's a bash script template with pattern substitution.

### Phase 2: PTY Watcher Pattern 8

**Deliverables:**
- `scope_violation` added to `ObservationCategory` in `pattern-matchers.ts`
- `matchScopeViolation` matcher implemented (receives `allowed_paths` from watcher context)
- Watcher construction accepts scope context from session metadata
- `scope_violation` events added to push notification triggers
- Unit tests with synthetic PTY output containing Write/Edit to in-scope and out-of-scope paths

**Why second:** The real-time detection layer depends on the metadata plumbing from Phase 1. Pattern 8 extends the existing watcher infrastructure (PRD 010) — the architecture is proven, this is a new matcher in the same pipeline.

### Phase 3: Commission Skill Integration

**Deliverables:**
- Commission skill template updated to include `allowed_paths` in spawn call instructions
- File-scope specification from Section 9 auto-mapped to glob patterns
- Documentation update: guide on scope enforcement for orchestrators

**Why third:** The commission skill is the adoption layer. Without it, orchestrators need to know about `allowed_paths` and remember to include it. The skill makes enforcement automatic for commissioned work.

## 6. Success Criteria

1. **Hard block works:** An agent spawned with `allowed_paths: ["packages/bridge/**"]` cannot commit changes to `packages/core/` — the pre-commit hook rejects the commit with a clear error message
2. **Real-time detection works:** When an agent writes to an out-of-scope file, a `scope_violation` event appears in `bridge_read_events` within 10 seconds
3. **Push notification works:** The parent agent receives a push notification when a child emits a `scope_violation` event
4. **Backwards compatible:** Sessions spawned without `allowed_paths` behave exactly as before — no hook, no scope checking
5. **Read is unrestricted:** Agents can read any file regardless of `allowed_paths` — scope constraints apply only to Write, Edit, and git commit
6. **Commission integration:** A `/commission` with file-scope specification produces a spawn call that includes `allowed_paths`
7. **Worktree fallback:** If `allowed_paths` is provided without `isolation: 'worktree'`, the bridge logs a warning and falls back to `mode: 'warn'` (no hook installed)
8. **No false positives:** Files matching `allowed_paths` patterns are never flagged as scope violations

## 7. Out of Scope

- **Runtime file-system sandboxing:** OS-level path restrictions (chroot, namespace isolation). The pre-commit hook and PTY watcher operate at the git and observation layers, not the filesystem layer. An agent can still write out-of-scope files — it just can't commit them (enforce mode) or does so with visibility (warn mode).
- **Dynamic scope expansion:** Allowing agents to request scope extensions at runtime ("I need to also modify X"). This is a future orchestration concern. For now, scope is fixed at spawn time.
- **Read restrictions:** Preventing agents from reading files. Agents need broad read access to understand context. Restricting reads would degrade work quality without meaningful security benefit (the agent is already trusted with the full codebase).
- **Network/command restrictions:** Limiting which bash commands or network endpoints the agent can use. This is a different enforcement domain (sandboxing) that goes beyond file-scope.
- **Cross-session scope coordination:** Detecting that two sessions have overlapping `allowed_paths` and might conflict. This is a future orchestration concern related to merge conflict prevention.
- **Scope enforcement for non-worktree sessions:** Pre-commit hooks require worktree isolation. Shared-mode sessions get warn-only behavior. Full enforcement in shared mode would require modifying the repository's actual hooks, which is unsafe.

## 9. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Bash glob matching diverges from gitignore semantics** — `**` and `*` don't behave identically in bash extglob vs gitignore | MEDIUM | MEDIUM | Use `git diff --cached --name-only` piped through pattern matching rather than pure bash globbing. Test with edge cases: nested dirs, dotfiles, paths with spaces. |
| **Hook prevents legitimate cross-cutting commits** (e.g., agent needs to update both code and docs) | MEDIUM | LOW | Orchestrator specifies all needed paths in `allowed_paths`. Commission skill's file-scope section already captures this. If the scope is wrong, the agent reports back and the orchestrator re-spawns with corrected scope. |
| **PTY watcher false positives on file paths** — Pattern 4's file path extraction may match paths in log output or error messages, not actual Write/Edit operations | LOW | LOW | Pattern 8 only fires on Write/Edit tool context (same as Pattern 4). Rate limiting and dedup prevent event storms from false positives. |
| **Pre-commit hook doesn't work on Windows** — bash hooks have portability issues on Windows git | MEDIUM | MEDIUM | Git for Windows ships with bash. Hooks in `.git/hooks/` work natively. Test on Windows (this project's primary platform). If bash fails, fall back to a Node.js hook script with a shebang to `node`. |
| **Agent circumvents hook** by staging and committing with `--no-verify` | LOW | HIGH | Claude Code's default behavior does not skip hooks. CLAUDE.md already instructs "Never skip hooks (--no-verify)". The PTY watcher (Pattern 8) provides a second detection layer for this case. |
| **Worktree hooks directory varies across git versions** — linked worktree hooks path is version-dependent | LOW | MEDIUM | Use `git rev-parse --git-path hooks/pre-commit` to resolve the correct hooks directory for the worktree, rather than hardcoding a path. |
