# Composition Runtime — CLM Execution Engine

Architecture spec for the Composed Language Model (CLM) execution engine.
Chains specialized SLMs with validation gates between stages, bounding error
propagation. See RFC 005 Parts III and VI for the theory.

## Overview

A CLM is a pipeline of stages, each either an SLM or a deterministic transform,
separated by validation gates:

```
Input → S₁ → V₁ → S₂ → V₂ → ... → Sₙ → Output
```

The runtime executes this pipeline, handles gate failures (retry/escalate),
and tracks per-stage metrics.

## Port Interfaces

### StagePort — What a Pipeline Stage Does

```typescript
/**
 * A single stage in a CLM pipeline. Either an SLM inference call,
 * a deterministic transform, or a sub-CLM.
 */
interface StagePort {
  readonly id: string;
  readonly type: 'slm' | 'deterministic' | 'clm';
  execute(input: StageInput): Promise<StageOutput>;
}

interface StageInput {
  /** The data to process. String for SLM text, structured for deterministic. */
  data: string;
  /** Metadata carried through the pipeline. */
  context: PipelineContext;
}

interface StageOutput {
  /** The stage's output data. */
  data: string;
  /** Confidence score (0-1). SLM stages report model confidence; deterministic = 1.0. */
  confidence: number;
  /** Latency of this stage in ms. */
  latencyMs: number;
}
```

### GatePort — What a Validation Gate Does

```typescript
/**
 * A validation gate between pipeline stages.
 * Returns pass/fail + reason. Pipeline runtime decides retry/escalate.
 */
interface GatePort {
  readonly id: string;
  validate(input: GateInput): Promise<GateResult>;
}

interface GateInput {
  /** The output from the preceding stage. */
  data: string;
  /** The expected format/schema (optional — some gates are format-agnostic). */
  schema?: unknown;
  /** Pipeline context for stateful gates. */
  context: PipelineContext;
}

interface GateResult {
  pass: boolean;
  /** Why it failed (for logging/debugging). */
  reason?: string;
  /** Parsed/validated data (gates may transform on pass). */
  validatedData?: unknown;
}
```

### InferencePort — How Stages Call SLMs

```typescript
/**
 * Backend-agnostic SLM inference. Wraps Ollama, ONNX, or HTTP bridge.
 * Reuses the existing SLMInference interface from phase-4-integration.
 */
interface InferencePort {
  readonly modelId: string;
  generate(input: string): Promise<{
    output: string;
    confidence: number;
    latencyMs: number;
  }>;
}
```

This maps directly to the existing `SLMInference` interface in
`experiments/exp-slm/phase-4-integration/src/slm-inference.ts`. Use the
Ollama adapter (`ollama-adapter.ts`) for chobits inference or the HTTP
bridge for local ONNX.

## Gate Types

### 1. ParseGate (Deterministic, Free)

Compiles a Peggy grammar and attempts to parse the stage output.

```
Input: stage output (string) + grammar (string)
Pass: output parses without error
Fail: parse error (with location + message)
Transform: returns parsed AST as validatedData
```

This is the primary gate for the Schema→Grammar pipeline. The grammar
is either provided statically or generated by a previous stage (B-1).

### 2. SchemaGate (Deterministic, Free)

Checks that parsed output matches an expected TypeScript type shape.

```
Input: parsed object + expected field names/types
Pass: all required fields present, correct types
Fail: missing field or type mismatch
```

### 3. ClassifierGate (SLM, Cheap)

An SLM that classifies output as valid/invalid. More expensive than
parse/schema gates but catches semantic errors that syntax can't detect.

```
Input: (stage input, stage output) pair
Output: valid | invalid | uncertain
Threshold: configurable (default: reject uncertain)
```

This is B-2 (Causal Validator) from RFC 005 — not yet built.

## Pipeline Definition

A pipeline is defined as an ordered list of stages and gates:

```typescript
interface PipelineDefinition {
  id: string;
  stages: PipelineStep[];
}

type PipelineStep =
  | { type: 'stage'; stage: StagePort }
  | { type: 'gate'; gate: GatePort; onFail: FailurePolicy };

interface FailurePolicy {
  /** Max retries before escalation. */
  maxRetries: number;
  /** Escalation strategy when retries exhausted. */
  escalation: 'abort' | 'skip' | 'frontier';
  /** Frontier LLM fallback config (if escalation = 'frontier'). */
  frontierConfig?: { provider: string; model: string; prompt: string };
}
```

### Example: Schema→Grammar Pipeline

```typescript
const schemaGrammarPipeline: PipelineDefinition = {
  id: 'schema-to-grammar',
  stages: [
    {
      type: 'stage',
      stage: b1SchemaGrammarSLM,      // SLM: TS interface → PEG grammar
    },
    {
      type: 'gate',
      gate: peggyCompileGate,          // Deterministic: does grammar compile?
      onFail: { maxRetries: 2, escalation: 'frontier' },
    },
    {
      type: 'stage',
      stage: exampleGeneratorStage,    // Deterministic: grammar → sample DSL
    },
    {
      type: 'gate',
      gate: peggyParseGate,            // Deterministic: does sample parse?
      onFail: { maxRetries: 1, escalation: 'abort' },
    },
  ],
};
```

## Execution Model

```
for each step in pipeline.stages:
  if step is 'stage':
    output = await step.stage.execute(currentInput)
    record metrics (latency, confidence)
    currentInput = output.data

  if step is 'gate':
    result = await step.gate.validate(currentInput)
    if result.pass:
      currentInput = result.validatedData ?? currentInput
      continue
    else:
      for attempt in 1..step.onFail.maxRetries:
        re-run previous stage
        result = await step.gate.validate(newOutput)
        if result.pass: break
      if still failing:
        apply escalation policy (abort / skip / frontier fallback)
```

### Retry Semantics

When a gate fails and retries the previous stage:
- SLM stages: re-run inference with `temperature > 0` to get different output
- Deterministic stages: retry is meaningless (same input → same output), skip to escalation

### Frontier Escalation

When retries are exhausted and `escalation = 'frontier'`:
- Call a frontier LLM (Claude) with the original input + the failed gate's error message
- The frontier LLM produces the output that the SLM couldn't
- Log the escalation for future training data (the frontier output becomes a training example)

This last point is critical: **every escalation is a training signal**. The
frontier LLM's output, paired with the original input, can be added to the
SLM's training corpus for the next fine-tuning run. Over time, escalations
decrease as the SLM learns from its failures.

## Metrics

### Per-Stage Metrics

```typescript
interface StageMetrics {
  stageId: string;
  latencyMs: number;
  confidence: number;
  retryCount: number;
  escalated: boolean;
  escalationTarget?: string;  // 'frontier' | 'skip'
}
```

### Pipeline Metrics

```typescript
interface PipelineMetrics {
  pipelineId: string;
  totalLatencyMs: number;
  stages: StageMetrics[];
  gatePassRate: number;       // gates passed / gates evaluated
  escalationRate: number;     // escalations / total runs
  endToEndSuccess: boolean;   // did the pipeline produce valid output?
}
```

### Aggregate Metrics (for Gate C-G1 and C-G2)

```typescript
interface AggregateMetrics {
  totalRuns: number;
  successRate: number;        // Gate C-G1: target >= 85%
  gateEffectiveness: number;  // Gate C-G2: error reduction vs ungated
  meanLatencyMs: number;
  escalationRate: number;
  costPerRun: number;         // SLM inference cost (effectively $0 for local)
}
```

## Inference Backend

For the initial implementation, use **Ollama on chobits** via HTTP:

```
SLM stage → HTTP POST http://chobits:11434/v1/chat/completions
  model: loaded from model path
  messages: [{ role: "user", content: stageInput }]
```

The `InferencePort` abstraction allows swapping to local ONNX later without
changing the pipeline logic.

For models not served by Ollama (fine-tuned LoRA models), use the
`generate-predictions.py` script pattern: load model directly with
transformers + torch. The composition runtime calls this via a thin HTTP
wrapper or subprocess.

## Validation Plan (Gates C-G1 and C-G2)

### Gate C-G1: End-to-End Accuracy

Run the Schema→Grammar pipeline on 50 novel TypeScript interfaces.
Success = final output is a grammar that compiles AND parses a sample.
**Target: >= 85%.**

### Gate C-G2: Gate Effectiveness

Ablation study:
1. Run pipeline WITH gates (retry + escalation)
2. Run pipeline WITHOUT gates (raw SLM output, no validation)
3. Compare error rates
**Target: gates reduce errors >= 50%.**

Given B-1 already achieves 100% compilability, C-G2 may require
intentionally degraded conditions (smaller corpus, fewer training steps)
to produce errors that gates can catch. Alternatively, test on harder
inputs (complex nested types, generics) where B-1 is more likely to fail.
